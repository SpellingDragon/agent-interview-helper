# 技术准备方案与题库优化计划 — 设计文档

## Context

项目当前仅依赖两份 Markdown 文件和一份零依赖 Python 脚本：

- `.personal/plan.md`：按 `## N. Day X：标题` 组织，内部使用 `### 今日目标 / 今日任务 / 今日输出 / 验收标准 / 晚间复盘`。
- `.personal/question.md`：按 `## N. 分类名` 组织，题目为 `### QN. 题目`，内容为 `#### 回答要点 / 一句话版 / 追问点 / 常见误区`。
- `interview_assistant.py`：读取上述结构，提供 `--topic`、`--practice-day`、`--mock` 等命令。

本轮优化只针对这两份 Markdown 的内容，目标是把它们从“可读的复习资料”升级为“可直接驱动高强度面试训练的资产”。所有方法论、参考资料、弱链追踪都通过文字内嵌到现有 section 中，不引入新文件，也不改代码。

## Goals / Non-Goals

**Goals:**
- 在 `.personal/plan.md` 中嵌入四层题模型和“回答四要素”方法论，并让每日任务、输出、复盘更具体。
- 在 `.personal/question.md` 中逐分类提升题目深度：每道核心题包含四要素回答、L3 场景追问、L4 身份/前沿追问、常见误区和项目锚点。
- 补充缺口题和跨域设计题，让 L3/L4 题占比明显提升。
- 在题目和计划中使用自然语言注明参考资料范围，形成“看哪里 → 练哪题 → 怎么答”的闭环。

**Non-Goals:**
- 不修改 `interview_assistant.py` 的解析逻辑、CLI 接口或状态文件格式。
- 不新增任何辅助 Markdown 文件（如 `domain-study-guide.md`、`prep-methodology.md`）。
- 不引入 YAML frontmatter、元数据 section 或任何新的标题格式。
- 不做通用语义检索、embedding 或自动化筛选。

## Decisions

### Decision 1: 方法论内嵌到现有章节

- **Rationale**: 新增独立的方法论文档会分散注意力，也违背“只优化两份文件”的目标。
- **做法**:
  - 在 `plan.md` 的 `1. 总原则` 中加入四层题模型和回答四要素说明。
  - 在 `question.md` 的 `1. 使用说明` 中加入题目质量标准（四要素、追问点、常见误区、项目锚定）。
- **Trade-off**: 文档开头变长，但读者打开任何一份文档都能看到标准，无需切换文件。

### Decision 2: 参考资料用自然语言注明

- **Rationale**: 不需要单独的映射表；在 `plan.md` 的 `今日任务` 和 `question.md` 的分类末尾用普通文本写出参考范围即可。
- **做法**:
  - `plan.md` 的每个 `今日任务` 末尾加一行：`参考资料：ai_infra_learning/docs/<domain>/...`。
  - `question.md` 的每个分类末尾加一个小节 `#### 本分类参考范围`，列出 2-5 个关键文件。
- **Trade-off**: 无法被机器索引，但满足“资料在哪里”的需求，且保持文档纯粹。

### Decision 3: 题目深度通过现有四个 section 实现

- **Rationale**: 现有结构已经被脚本稳定解析，任何新增 section 都会增加复杂度和风险。
- **做法**:
  - `#### 回答要点` 必须按四要素组织：定义/机制 → 为什么重要 → 工程做法 → 权衡/边界/失败模式。
  - `#### 一句话版` 保持 30 秒结论。
  - `#### 追问点` 至少包含一条 L3 场景追问和一条 L4 身份/前沿追问。
  - `#### 常见误区` 指出一个浅答案或错误。
- **Trade-off**: 题目篇幅变长，但深度和可训练性显著提升。

### Decision 4: 项目锚定融入内容，不单独标注

- **Rationale**: 项目锚定是答案质量的一部分，不需要元数据标签。
- **做法**: 在 `回答要点` 和 `追问点` 中自然引用 `长程 Agent 框架`、数据流水线 Agent、策略平台 / 规则引擎 / MCP Tool 的具体设计或踩坑经历。
- **Trade-off**: 需要人工确保每道核心题都有锚定，但文档更自然。

### Decision 5: Plan.md 的每日任务直接引用 qid

- **Rationale**: 让计划与题库形成 executable 的闭环。
- **做法**: 在 `今日任务` 中写明“练习 Q5-Q7”或“复习 Memory 分类 Q28-Q36”，而不是泛泛写“复习 Memory”。
- **Trade-off**: 计划需要随题库编号变化而维护，但训练更聚焦。

### Decision 6: 不拆分或新增 Day，只在现有 Day 中增强内容

- **Rationale**: 保持 14 天框架稳定，减少认知负担。
- **做法**: 每个 Day 的 `今日任务`、`今日输出`、`晚间复盘` 内容增强，但 Day 数量和标题不变。
- **Trade-off**: 某些 Day 内容会变多，但结构不变，脚本解析无影响。

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| 题目篇幅过长，练习负担重 | 每道题优先补全“追问点”和“常见误区”；回答要点按四要素精炼，不展开成长文。 |
| 计划引用 qid 后题库编号变动导致不一致 | 每次调整 `question.md` 编号后，全局搜索 `plan.md` 中的 qid 并同步。 |
| 过度追求深度，基础题变少 | L1/L2 题保留“一句话版”快答能力，只在追问点中升级深度。 |
| 参考资料写死路径后文档更新失效 | 每季度随 `ai_infra_learning/docs` 更新做一次全局审阅。 |
| 不新增文件导致方法论和资料索引混杂 | 在 `plan.md` 和 `question.md` 开头用清晰的二级/三级标题隔离方法论说明。 |

## Migration Plan

1. 备份 `.personal/question.md` 和 `.personal/plan.md`。
2. 在 `plan.md` 的 `1. 总原则` 中写入四层题模型和回答四要素。
3. 在 `question.md` 的 `1. 使用说明` 中写入题目质量标准。
4. 按弱链和领域优先级选择一个分类开始优化：
   - 重写该分类下每道题的 `回答要点`，补充 `追问点` 和 `常见误区`。
   - 在该分类末尾补充 `本分类参考范围`。
   - 在 `plan.md` 对应 Day 的 `今日任务` 中引用这些题目的 qid 和参考资料。
5. 完成一个分类后，用 `interview_assistant.py --topic <关键词>` 验证格式和练习体验。
6. 循环直到所有分类完成。
7. 补充 5-10 道跨域 L4 题到合适分类。
8. 更新 `plan.md` 末尾的最终检查表，确保覆盖 L1-L4。
9. 运行一次完整 mock，根据结果微调。

## Open Questions

- 目标岗位级别是 senior/staff 还是其他？这会决定 L4 题比重和系统设计深度。
- 当前最弱的分类是哪一个？建议从该分类启动第一轮优化。
- 是否允许在某些分类末尾使用 `#### 本分类参考范围`？这符合现有解析器对四级标题的收集规则。
- 是否需要把 目标团队 具体技术（MLA、GRPO、DualPipe、Engram、Mooncake）作为独立小分类，还是分散到现有分类中？
