# Agent 平台 / AI Infra 高频问答题库

> 适用对象：当前会话中的候选人背景
> 目标：把面试中的高频问题准备成可直接口述的题库
> 使用方式：
> 1. 先自己口头回答题目
> 2. 再看“回答要点”检查是否漏项
> 3. 最后补成 1 分钟版和 3 分钟版

## 1. 使用说明

建议每道题至少准备三层表达：

- `30 秒版`：先给结论，适合快问快答
- `2 分钟版`：给定义、场景、权衡
- `追问版`：补充失败模式、工程细节、 trade-off

回答时按“四要素”组织：

1. `定义/机制`：先讲清楚它是什么
2. `为什么重要`：说明在真实系统中的作用
3. `工程做法`：常见实现或自己的实践经验
4. `权衡/边界/失败模式`：什么时候会崩、怎么做取舍

对于 L2-L4 题，还要做到：

- `项目锚定`：至少结合 `长程 Agent 框架`、数据流水线 Agent、策略平台 / 规则引擎 / MCP Tool 中的一个
- `追问点`：每道题准备至少一条 L3 场景追问和一条 L4 身份/前沿追问
- `常见误区`：明确一个容易被面试官反驳的浅答案

L1-L2 题以“快准稳”为主，保持 30 秒版独立；L3-L4 题重点练追问版和项目锚定。

## 2. 岗位匹配与动机

### Q1. 你为什么适合 Agent 平台 / AI Infra 团队？

#### 回答要点

- **定义/机制**：Agent Runtime 的核心是把模型能力稳定地转化为可执行、可观测、可回滚的任务完成能力，需要运行时抽象、上下文管理、工具系统、记忆与 Eval 的交叉经验。
- **为什么重要**：这个岗位不是接 API 写 prompt，而是设计能承载复杂、长程、多轮 Agent 行为的工程系统。
- **工程做法**：过去做规则引擎、DAG Runtime、DSL、SDK 和实验配置，本质上已经在处理“把不稳定策略变成可配置、可灰度、可回滚的系统”；`长程 Agent 框架`、数据流水线 Agent、策略平台 MCP Tool 又补上了 Agent 运行时、Memory 管理和 Trace 的实践。
- **权衡/边界**：优势是能把复杂系统经验和 Agent 新范式结合；潜在短板是需要证明对模型底层（KV Cache、推理成本、后训练）有足够理解，而不是只停留在上层封装。

#### 一句话版

我适合这个岗位，不是因为我刚开始接触 Agent，而是因为我本来就在做复杂运行时和策略系统，现在只是把这套能力迁移到 Agent Runtime。

#### 追问点

- **L3 场景**：如果面试官质疑“你过去做的是传统业务策略系统，和 Agent Runtime 的模型行为策略平台差别很大”，你怎么回应？
- **L4 身份/前沿**：目标团队 的推理模型在 Agent 长程任务里成本容易爆炸，你在 `长程 Agent 框架` 或数据流水线 Agent 里是怎么考虑上下文成本的？

#### 常见误区

- 只说“我对 AI 很有热情”或“我学习能力强”，没有具体项目支撑。

### Q2. 你为什么想做 Agent Runtime，而不是纯模型或纯应用？

#### 回答要点

- **定义/机制**：纯模型研究能力边界，纯应用关注业务闭环，Agent Runtime 是连接两者的运行时层。
- **为什么重要**：模型能力再强，也需要工程系统把它稳定地转化为可复用、可观测、可迭代的任务执行能力。
- **工程做法**：自己最擅长的是把不稳定、快速变化的能力组织成可配置、可观测、可灰度、可回滚的系统。`长程 Agent 框架` 里的 EventKey / Memory 设计、策略平台 / 规则引擎里的配置热发布，都是同一类问题。
- **权衡/边界**：纯模型岗位离工程落地远，纯应用岗位又容易变成业务胶水；Agent Runtime 正好能让我发挥系统抽象和跨层调试能力。

#### 一句话版

我想做 Agent Runtime，因为它处在模型能力和真实任务之间，正好是我擅长的“把不稳定能力工程化”的方向。

#### 追问点

- **L3 场景**：如果让你选，你更愿意在 Agent Runtime 里解决“上下文组织”还是“工具调用稳定性”？为什么？
- **L4 身份/前沿**：你怎么看 目标团队 的 Mooncake / DualPipe 这类推理基础设施对 Agent Runtime 设计的影响？

#### 常见误区

- 回答成“模型太深奥我不行，应用太业务我不爱”，显得没有主见。

### Q3. 你理解的 Agent Runtime 是什么？

#### 回答要点

- **定义/机制**：Agent Runtime 不只是一个 Agent SDK，而是把模型能力稳定转化为真实任务执行能力的运行时和工程系统。
- **为什么重要**：没有 Agent Runtime，模型只能“回答”；有了 Agent Runtime，模型才能“完成任务”，并且这个过程需要可观测、可调试、可迭代。
- **工程做法**：它至少要解决五类问题：上下文管理、工具调用、状态与记忆、执行可观测、反馈闭环。`长程 Agent 框架` 的 Memory/EventKey、数据流水线 Agent的 Tool 编排、策略平台 / 规则引擎的配置化策略，分别对应了其中的不同部分。
- **权衡/边界**：Agent Runtime 不是越多越好；过度抽象会增加延迟和调试成本，设计时要平衡灵活性与可控性。

#### 一句话版

Agent Runtime 是让模型从“能回答”变成“能稳定完成任务”的运行时和工程系统。

#### 追问点

- **L3 场景**：如果一个 Agent Runtime 系统里模型频繁选错 Tool，你会优先改模型、改 Prompt，还是改工具 schema？
- **L4 身份/前沿**：对比 AutoGPT 早期 Demo 和 目标团队 可能需要的 Agent Runtime，你认为关键差异在哪？

#### 常见误区

- 把 Agent Runtime 简单等同于“Agent 框架”或“工作流引擎”，忽略运行时和反馈闭环。

### Q4. 你为什么认为这个岗位需要广度而不是单点技能？

#### 回答要点

- **定义/机制**：Agent Runtime 横跨模型、运行时、工具系统、后端基础、开发者体验和产品迭代。
- **为什么重要**：真实问题往往不是单点 bug，而是上下文、工具、模型和策略耦合后的综合问题。
- **工程做法**：在 `长程 Agent 框架` 里同时涉及 Go 运行时、Memory 设计、Tool schema、Trace；在数据流水线 Agent 里同时涉及多模态理解、视频处理、投稿链路；这些都需要跨层理解。
- **权衡/边界**：广度不等于浅度，关键是能在不同层之间定位问题：是模型问题、上下文问题、工具问题还是基础设施问题。

#### 一句话版

Agent Runtime 的问题天然跨层，单点技能很难定位“模型选错 Tool 是因为 schema 不清晰还是上下文被污染”。

#### 追问点

- **L3 场景**：如果团队里研究员认为某个任务模型搞不定，是模型问题，你怎么判断其实是 Agent Runtime 的上下文设计问题？
- **L4 身份/前沿**：在你的项目里，哪一次失败最典型地体现了“跨层耦合”而不是单点 bug？

#### 常见误区

- 把广度当成“我什么都懂一点”，没有举出跨层协作或定位问题的具体例子。

## 3. 项目与经历

### Q5. 你做的 `长程 Agent 框架` 长程 Agent 框架，核心探索点是什么？

#### 回答要点

- **定义/机制**：`长程 Agent 框架` 是基于 Go 和 `trpc-agent-go` 的轻量级长程 Agent 框架，核心不是再造一个聊天壳，而是探索长任务中的 Memory、上下文注入、多 Agent 协作和执行可观测。
- **为什么重要**：长程 Agent 如果直接堆上下文，token 成本、延迟和调试复杂度都会爆炸，需要把“上下文管理”从模型侧搬到工程侧。
- **工程做法**：重点抽象 Agent、Model、Tool、Skill、Memory、Trace；Agent 间不传大段上下文而传 `EventKey`，由 runtime 按任务阶段按需装配真实上下文。
- **权衡/边界**：好处是控制上下文膨胀、降低重复传输、提升可维护性；代价是引入了一层 indirection，调试时要能追踪 key 到真实内容的映射。

#### 一句话版

`长程 Agent 框架` 的核心探索点是：在长程 Agent 任务里，用 Memory + EventKey 把上下文管理从模型侧搬到工程侧，从而控制成本、提升可观测性。

#### 追问点

- **L3 场景**：如果 `EventKey` 设计导致延迟抖动，你会怎么定位？是 key 解析慢、Memory 读取慢，还是上下文装配本身太大？
- **L4 身份/前沿**：若把 目标团队 MLA 的 KV 压缩思想应用到 `长程 Agent 框架` 的 Memory，你会怎么设计 key 的存储与读取策略？

#### 常见误区

- 只讲“我做了 Agent 框架”，不解释为什么需要 EventKey、Memory 这些抽象，以及解决了什么真实痛点。

### Q6. 为什么 Agent 间不直接传上下文，而是传 `EventKey`？

#### 回答要点

- **定义/机制**：`EventKey` 是一个轻量级引用，Agent 之间通过 key 传递“需要哪段上下文”的信号，真实上下文由 runtime 在需要的时候再装配。
- **为什么重要**：大段上下文直接传递会带来 token 成本、重复信息、历史污染和角色耦合问题；长程任务里这些问题会被放大。
- **工程做法**：在 `长程 Agent 框架` 中，不同阶段真正需要的信息不同，runtime 按任务阶段选择性注入；Agent 只感知 key，真实上下文由工程侧管理，更利于裁剪、审计、复用和调试。
- **权衡/边界**：换来了可控性和成本优势，但增加了 indirection，错误排查时要能追踪 key 到真实内容的链路；key 引用失效时需要兜底策略。

#### 一句话版

传 `EventKey` 而不是传全文，是为了把“上下文引用”和“上下文内容”解耦，从而控制成本、隔离噪声并统一由 runtime 管理。

#### 追问点

- **L3 场景**：如果一个 `EventKey` 引用了一段已经被清理的 Memory，Agent 行为会出什么问题？你怎么设计兜底？
- **L4 身份/前沿**：在数据流水线 Agent 这种长链路系统里，是否也适合引入 `EventKey` 思想？哪些阶段适合、哪些阶段不适合？

#### 常见误区

- 只说“为了省 token”，没讲清楚解耦后对调试、审计、权限控制的价值。

### Q7. `长程 Agent 框架` 中 Skill 和 Tool 的区别是什么？

#### 回答要点

- **定义/机制**：Tool 是原子能力（查询、上传、解析、执行）；Skill 是可复用的任务模板或策略块，通常会组合多步 Tool、约束调用方式或定义角色边界。
- **为什么重要**：Tool 解决“能做什么”，Skill 解决“这类任务通常怎么做”。没有 Skill 层，模型每次都要重新规划常见任务，成本高且不稳定。
- **工程做法**：在 `长程 Agent 框架` 中，Skill 可以携带默认的调用偏好、重试策略和上下文装配规则；Tool 只负责执行并返回结构化结果。
- **权衡/边界**：Skill 层过厚会变成隐式工作流，降低灵活性；过薄则没有沉淀价值。关键是让 Skill 可观测、可版本化、可 A/B 测试。

#### 一句话版

Tool 是“能做什么”的原子能力，Skill 是“这类任务通常怎么做”的可复用策略块。

#### 追问点

- **L3 场景**：如果某个 Skill 频繁调用同一个 Tool 导致下游被打爆，你会在 Skill 层做限流，还是在 Tool 层做限流？为什么？
- **L4 身份/前沿**：策略平台 / 规则引擎里的“策略块”和 `长程 Agent 框架` 里的 Skill 有什么异同？能否把规则引擎的 DSL 经验迁移到 Skill 描述上？

#### 常见误区

- 把 Skill 和 Tool 混为一谈，或者把 Skill 描述成“一个函数”而忽略策略和约束语义。

### Q8. 数据流水线 Agent 中，哪些步骤是 Agent 决策，哪些是工程固定？

#### 回答要点

- **定义/机制**：把链路拆分为“确定性步骤”和“不确定性步骤”。事件监听、直播流获取、ffmpeg 处理、投稿接口调用等偏工程和确定性步骤，固定在 Tool 或工作流中；哪些互动片段值得分析、如何组合已有工具、是否需要做多模态理解、更适合哪种切片方案，由 Agent 规划。
- **为什么重要**：如果把所有步骤都交给模型，成本高、延迟大、结果不可控；如果全部工程固定，又失去了 Agent 的灵活性和语义理解能力。
- **工程做法**：数据流水线 Agent 中，工程层负责可靠执行和数据兜底，Agent 层负责语义判断和策略选择，两者通过 Tool schema 和 Memory key 交互。
- **权衡/边界**：边界不是一成不变的；随着模型能力增强，部分原本由工程固定的步骤可以逐步交给 Agent，但需要 Eval 和灰度保护。

#### 一句话版

确定性步骤（采集、处理、投稿）交给工程，不确定性步骤（选片段、定切片方案）交给 Agent。

#### 追问点

- **L3 场景**：20 分钟直播链路里，如果模型把“弹幕起哄”误判成“高光时刻”，你的 Trace 和反馈闭环怎么修？
- **L4 身份/前沿**：如果未来模型对视频语义理解更强，你会把“切片方案生成”从 Agent 决策进一步下沉到端到端模型，还是保持 Agent 编排？

#### 常见误区

- 认为“所有步骤都应该是 Agent 决策”或“Agent 只是调用外部 API 的壳”。

### Q9. 数据流水线 Agent 最难的工程问题是什么？

#### 回答要点

- **定义/机制**：数据流水线 Agent 是一个长链路自动化系统，包含事件采集、互动抽样、热点识别、多模态理解、切片方案生成、视频处理、投稿和归档。
- **为什么重要**：链路越长，中间状态越多，失败点越多；同时直播是实时流，不能简单重跑整个链路。
- **工程做法**：难点有三：1）中间状态管理复杂，需要明确每个步骤的输入输出和回滚策略；2）上下文对象很大，不能直接塞给模型，需要做 memory key 分离和 Tool 侧取数；3）多模态链路耗时长、失败点多，需要 Trace 和可重试设计。
- **权衡/边界**：不能为了可观测性记录所有中间结果（存储和隐私成本），需要定义最小可回溯集合；也不能为了速度牺牲切片质量。

#### 一句话版

数据流水线 Agent 最难的是长链路中间状态管理、大上下文不塞模型、多模态链路失败恢复这三件事的叠加。

#### 追问点

- **L3 场景**：如果投稿接口在某一步失败，前面已经生成的视频片段怎么处理？是重试、跳过还是回滚？
- **L4 身份/前沿**：这个长链路系统如果要用 目标团队 的 Mooncake 做 PD 分离加速多模态推理，你会把哪几步放到 Prefill 节点、哪几步放到 Decode 节点？

#### 常见误区

- 只讲“链路长”，没有讲清楚具体哪个状态最难管、为什么难、怎么解决。

### Q10. 策略平台 / 规则引擎和 Agent Runtime 有什么关系？

#### 回答要点

- **定义/机制**：规则引擎的对象是“业务策略”，Agent Runtime 的对象是“模型驱动行为”，但两者都要解决抽象表达、运行时执行、灰度回滚、校验、可观测、实验闭环。
- **为什么重要**：模型行为（prompt、tool 选择、采样参数）也需要像业务策略一样快速迭代、安全发布、可回滚。
- **工程做法**：策略平台项目里把模型、prompt、tool 的策略从代码迭代抽成配置热变更，这本质上已经在做 Agent Runtime 的一部分：把不稳定的能力通过配置化运行时管理起来。
- **权衡/边界**：规则引擎通常处理确定性策略，Agent Runtime 还要处理模型的概率性输出；所以 Agent Runtime 更需要 Eval、Trace 和反馈闭环。

#### 一句话版

策略平台 / 规则引擎是从“业务策略配置化”向“模型行为配置化”的延伸，和 Agent Runtime 共享运行时、灰度、可观测这些核心抽象。

#### 追问点

- **L3 场景**：如果一条 prompt 策略配置错误导致线上模型输出异常，你的回滚流程是什么样的？多久能恢复？
- **L4 身份/前沿**：如果让 `长程 Agent 框架` 的 Skill 也接入策略平台 / 规则引擎的配置热变更能力，你会怎么设计配置 schema 和版本管理？

#### 常见误区

- 只说“都是策略系统”，没讲清楚模型行为的概率性给 Agent Runtime 带来的额外挑战。

### Q11. 为什么把模型策略从代码发布改成配置热变更？

#### 回答要点

- **定义/机制**：把模型、prompt、tool 策略从硬编码发布改为通过配置平台热变更。
- **为什么重要**：模型能力的迭代速度远快于传统业务逻辑，如果每次调 prompt 都要走代码发布，实验效率会被严重拖累。
- **工程做法**：配置平台和实验平台、审批、回滚、灰度体系融合；SDK 本地预解析配置以提升热点路径性能；配置变更经过校验和审批后才能生效。
- **权衡/边界**：配置化带来灵活性，但也引入“错误配置直接上线”的风险，必须配套 schema 校验、灰度发布和一键回滚。

#### 一句话版

改成配置热变更，是为了让模型策略的迭代速度和实验闭环追上模型能力本身的迭代速度。

#### 追问点

- **L3 场景**：配置热变更时，如何保证新旧版本配置的平滑切换？如果灰度过程中发现异常，如何快速全量回滚？
- **L4 身份/前沿**：Agent Runtime 里的 Prompt/Policy 管理如果也做成配置热变更，你认为它和策略平台 / 规则引擎最大的差异会是什么？

#### 常见误区

- 只讲“发布快”，不讲清楚审批、校验、回滚这些安全底线。

### Q12. 在线 Agent 内容解析 Tool 为什么以 MCP 形式提供？

#### 回答要点

- **定义/机制**：MCP 把多媒体资源解析能力标准化为 Tool 协议，任何宿主或上层 Agent 都可以按统一 schema 调用。
- **为什么重要**：内容解析能力（图片、音频、视频、网页）具有通用性，如果嵌死在单一业务服务里，会被重复实现、难以复用和治理。
- **工程做法**：以 MCP 形式提供后，可以统一做参数校验、权限控制、缓存、限流、Trace 和版本管理；策略平台内容解析 Tool 就可以被多个 Agent 复用。
- **权衡/边界**：MCP 增加了协议转换和治理成本；只有当能力通用、边界清晰、需要多宿主复用时才适合 MCP 化。

#### 一句话版

以 MCP 形式提供，是为了把通用内容解析能力标准化、可复用、可治理，而不是嵌死在单一业务服务里。

#### 追问点

- **L3 场景**：如果多个 Agent 同时高频调用这个 MCP Tool，你会如何保护下游解析服务？
- **L4 身份/前沿**：对比 MCP 和 Function Call，你认为在 Agent Runtime 这种多 Agent 环境里，MCP 的核心价值是协议统一还是治理边界？

#### 常见误区

- 把 MCP 当成“远程函数调用”，忽略它在权限、缓存、观测、版本治理上的价值。

## 4. Agent / Agent Runtime 核心

### Q13. ReAct 是什么？为什么工程上常用它？

#### 回答要点

- **定义/机制**：ReAct 是模型在推理（Reasoning）和行动（Acting）之间交替的一种模式，典型链路是“思考 → 选择工具 → 执行 → 观察结果 → 再思考”。
- **为什么重要**：很多任务无法靠一次模型调用完成，需要借助外部信息或工具；ReAct 把推理过程显式化，让模型一步一步逼近目标。
- **工程做法**：在 `长程 Agent 框架` 中，可以把 ReAct 的每一步映射到 Trace 里，记录思考、工具调用参数、返回结果和状态变更，方便调试和回放。
- **权衡/边界**：ReAct 增加了调用轮数和延迟；如果任务流程高度确定，直接用工程工作流会更便宜、更可控。

#### 一句话版

ReAct 是“思考-行动-观察”交替的推理模式，工程上常用是因为每一步都可观测、可约束、可调试。

#### 追问点

- **L3 场景**：在 ReAct 循环里，如果某一步工具调用返回了错误但模型却认为结果正确并继续执行，你会怎么发现和兜底？
- **L4 身份/前沿**：目标团队 推理模型的 CoT 能力变强后，ReAct 的“思考”步骤是否还需要显式暴露给工程侧？

#### 常见误区

- 把 ReAct 当成万能模式，对确定性任务也硬套循环，导致不必要的延迟和成本。

### Q14. Agent Loop 的核心状态有哪些？

#### 回答要点

- **定义/机制**：Agent Loop 是任务驱动下的循环执行结构，核心状态包括：接收任务、生成下一步决策、选择/调用工具、获取观察结果、更新状态与上下文、判断完成或进入异常处理。
- **为什么重要**：状态划分清晰后，才能做可观测、可重试、可回滚的 Agent Runtime。
- **工程做法**：在 `长程 Agent 框架` 中，这些状态通过 Trace 和 Memory 持久化；任何一步失败都可以从上一个稳定状态恢复，而不是从头重跑。
- **权衡/边界**：状态切分太细会增加存储和调试复杂度；太粗则不利于失败恢复和细粒度观测。

#### 一句话版

Agent Loop 的核心状态是：接收任务 → 决策 → 调用工具 → 观察 → 更新状态 → 判断终止/异常。

#### 追问点

- **L3 场景**：如果 Tool 调用超时，Loop 应该重试、跳过还是转人工？不同选择对状态机有什么要求？
- **L4 身份/前沿**：数据流水线 Agent 的长链路里，哪些步骤的状态必须持久化，哪些可以放在内存里？

#### 常见误区

- 只列出状态，不讲状态之间如何转换、失败如何恢复。

### Q15. 为什么说 Agent 不等于多轮聊天？

#### 回答要点

- **定义/机制**：多轮聊天只是连续对话，未必涉及外部动作和状态管理；Agent 需要任务驱动、工具执行、状态演进、失败恢复和目标达成。
- **为什么重要**：如果把 Agent 做成聊天壳，就无法完成“订机票”、“写代码”、“处理视频”等需要外部动作的任务。
- **工程做法**：`长程 Agent 框架` 的重点不是对话体验，而是长任务中的 Memory、Tool 调用和 Trace；数据流水线 Agent 则需要驱动 ffmpeg、投稿接口等外部系统。
- **权衡/边界**：对话可以是 Agent 的交互形式之一，但 Agent 的核心是闭环执行能力，而不是轮数多少。

#### 一句话版

Agent 与多轮聊天的关键区别是：Agent 要完成任务，涉及工具执行、状态管理和失败恢复，而不只是连续对话。

#### 追问点

- **L3 场景**：如果一个系统看起来像多轮聊天，但偶尔会调用工具查天气，它是 Agent 吗？你判断的标准是什么？
- **L4 身份/前沿**：在 Agent Runtime 设计里，你会把“对话管理”和“任务执行”放在同一层还是分离成两层？

#### 常见误区

- 把 Agent 简单等同于“能联网的聊天机器人”。

### Q16. Tool Use 的完整流程是什么？

#### 回答要点

- **定义/机制**：Tool Use 是模型根据目标选择并调用外部能力的流程：声明可用工具及参数 schema → 模型选择工具 → 生成结构化参数 → 工程侧反序列化/校验 → 执行工具 → 获取结果 → 回填给模型 → 继续决策或终止。
- **为什么重要**：Tool Use 是模型从“回答”走向“行动”的关键机制，也是 Agent Runtime 的核心能力之一。
- **工程做法**：在 `长程 Agent 框架` 中，Tool schema 是模型和工程系统之间的契约；执行结果会经过裁剪和格式化后再回填，避免污染上下文。
- **权衡/边界**：schema 越严格，模型越稳定但灵活性越低；执行结果越长，上下文成本越高，需要摘要或 key 引用。

#### 一句话版

Tool Use 的完整流程是：声明 schema → 模型选工具 → 生成参数 → 工程校验执行 → 获取结果 → 回填继续决策。

#### 追问点

- **L3 场景**：如果模型生成的参数格式合法但业务上不可能执行（例如查询不存在的订单号），你应该让 Tool 报错还是让模型自己从结果里推理？
- **L4 身份/前沿**：策略平台 MCP Tool 的内容解析流程里，如果模型传了一个超大 URL 列表，你会在 Tool 层还是 Agent 层做分页控制？

#### 常见误区

- 认为 Tool Use 只是“模型调用 API”，忽略 schema 设计、参数校验、错误映射和结果裁剪。

### Q17. Tool schema 为什么重要？

#### 回答要点

- **定义/机制**：Tool schema 是模型和工程系统之间的契约，描述了工具能做什么、需要什么参数、返回什么结构。
- **为什么重要**：schema 越清晰，模型越容易稳定地产出正确参数；工程侧也能据此做校验、错误提示、版本兼容和调试。
- **工程做法**：在 `长程 Agent 框架` 中，好的 schema 会包含参数类型、必填项、取值范围、示例和错误说明；策略平台 MCP Tool 的 schema 还要考虑多媒体输入的边界。
- **权衡/边界**：schema 太灵活容易让模型幻觉；太死板会限制使用场景。需要持续根据线上调用数据迭代。

#### 一句话版

Tool schema 是模型和工程系统的契约，决定模型能否稳定、正确地调用工具。

#### 追问点

- **L3 场景**：如果模型频繁把“查询订单”和“取消订单”两个 Tool 搞混，你优先改 schema 描述、改 Prompt，还是合并 Tool？
- **L4 身份/前沿**：对于 目标团队 这种强推理模型，未来是否还需要手工写详细 schema，还是可以让模型从文档中自动生成？

#### 常见误区

- 认为 schema 只是“给模型看的”，忽略它对校验、版本治理和错误恢复的价值。

### Q18. Tool 设计最常见的失败模式有哪些？

#### 回答要点

- **定义/机制**：Tool 设计失败模式是指工具在 schema、执行、返回或治理上容易出的问题。
- **为什么重要**：Tool 一旦设计不好，会直接影响 Agent 的稳定性、成本和用户体验。
- **工程做法**：常见失败模式包括：模型选错工具、参数缺失或字段幻觉、非幂等工具被错误重试、返回结果过大污染上下文、工具内部下游超时或失败、工具结果正确但不符合用户真正目标。
- **权衡/边界**：防御这些失败需要 schema、Prompt、工程校验、重试策略、结果裁剪和 Eval 共同作用，不能只靠一层。

#### 一句话版

Tool 设计最常见的失败模式包括：选错工具、参数幻觉、非幂等重试、返回过大、下游失败、结果与目标偏离。

#### 追问点

- **L3 场景**：如果一个非幂等 Tool 因为网络超时被重试，导致用户被扣了两次款，你会怎么从 schema 和重试策略上避免？
- **L4 身份/前沿**：在 `长程 Agent 框架` 里，你会把“幂等性检查”放在 Tool 实现里，还是放在 Agent runtime 的调用层？

#### 常见误区

- 只关注 Tool 功能是否正确，忽略 Tool 在 Agent loop 里的调用模式、失败传播和上下文影响。

### Q19. Tool 返回太长怎么办？

#### 回答要点

- **定义/机制**：Tool 返回太长会直接撑大上下文，增加 token 成本、延迟和噪声，甚至导致模型注意力分散。
- **为什么重要**：Agent 多轮 loop 中，如果每次 Tool 返回都全量回填，上下文会指数级膨胀。
- **工程做法**：不要把全量原始结果直接回填模型；优先做结构化摘要、分层返回、分页、ID 引用或 key 引用；必要时只返回可供下一步决策的最小信息集。`长程 Agent 框架` 的 EventKey 思想在这里同样适用。
- **权衡/边界**：摘要过多会丢失细节，导致模型决策错误；需要保留原始数据的访问路径，方便模型在必要时追问。

#### 一句话版

Tool 返回太长时，要做摘要、分页、ID 引用或 key 引用，只把决策所需的最小信息回填模型。

#### 追问点

- **L3 场景**：如果一个数据库查询 Tool 返回了 1000 条记录，模型下一步要“找出最近 7 天金额最大的 3 条”，你会让 Tool 在服务端做完聚合，还是把 1000 条塞给模型？
- **L4 身份/前沿**：在数据流水线 Agent 里，多模态理解 Tool 返回的视频特征向量很大，你会怎么设计返回格式以控制上下文？

#### 常见误区

- 直接把 Tool 原始输出全量回填，导致上下文爆炸和模型注意力分散。

### Q20. Skill 和 Workflow 有什么区别？

#### 回答要点

- **定义/机制**：Workflow 强调流程和依赖，通常由工程预定义；Skill 是任务模式的封装，可以包含半结构化策略、调用偏好和约束。
- **为什么重要**：Workflow 适合确定性流程，Skill 适合需要模型灵活决策但又有经验可循的任务。
- **工程做法**：在 `长程 Agent 框架` 中，Skill 可以定义默认 Tool 组合、重试策略和上下文装配规则；Workflow 则用于数据流水线 Agent 中“下载 → 转码 → 投稿”这类固定链路。
- **权衡/边界**：Skill 层过厚会变成隐式工作流，降低灵活性；Workflow 过硬则无法利用模型的语义理解能力。

#### 一句话版

Workflow 是工程预定义的确定性流程，Skill 是可复用的任务模式封装，更贴近“这类任务通常怎么做”。

#### 追问点

- **L3 场景**：如果一个任务 80% 步骤固定、20% 需要模型判断，你会做成 Workflow + Skill 的混合，还是纯 Skill？
- **L4 身份/前沿**：策略平台 / 规则引擎里的策略配置和 `长程 Agent 框架` 的 Skill，在抽象层级上有什么对应关系？

#### 常见误区

- 把 Skill 当成“一个函数”或“一个固定流程”，忽略它的策略和约束语义。

### Q21. 什么情况下应该做 Multi-Agent？

#### 回答要点

- **定义/机制**：Multi-Agent 是把任务拆分成多个子任务，由不同 Agent 分别负责，并通过协议协作完成。
- **为什么重要**：当单 Agent 的上下文过大、角色冲突明显、工具边界不同或子任务需要独立评估时，拆分可以降低复杂度和成本。
- **工程做法**：在 `长程 Agent 框架` 中，Multi-Agent 拆分会同时考虑上下文隔离、工具边界和独立 Trace；数据流水线 Agent 的“内容理解”和“切片生成”理论上可以拆给不同 Agent。
- **权衡/边界**：Multi-Agent 增加了通信、协议和调试成本；默认先从单 Agent 做起，除非任务分解天然成立。

#### 一句话版

当任务可清晰拆分、子任务需要不同能力边界、上下文隔离收益大于协作成本时，才应该做 Multi-Agent。

#### 追问点

- **L3 场景**：如果两个 Agent 同时操作同一个 Memory，一个要读、一个要写，你会怎么设计并发控制？
- **L4 身份/前沿**：目标团队 的推理模型上下文窗口越来越大，是否意味着 Multi-Agent 的必要性在降低？你怎么看？

#### 常见误区

- 为了 Multi-Agent 而拆分，增加不必要的通信成本和调试难度。

### Q22. 为什么很多 Multi-Agent 系统效果并不好？

#### 回答要点

- **定义/机制**：Multi-Agent 效果不好的根本原因是拆分了角色但没有真正的隔离和专业化收益。
- **为什么重要**：如果子 Agent 之间信息共享不清晰、目标不一致，协作成本会超过收益。
- **工程做法**：常见失败原因包括：机械分角色没有信息隔离、通信和 token 成本显著上升、中间状态和协议设计复杂、错误在子 Agent 间传递放大、缺乏独立 Eval。
- **权衡/边界**：Multi-Agent 不是银弹，关键是拆分时是否有明确的边界、评估标准和协作协议。

#### 一句话版

很多 Multi-Agent 系统效果不好，是因为只拆角色没拆边界，导致通信成本高、调试难、错误会放大。

#### 追问点

- **L3 场景**：如果一个 Multi-Agent 系统里两个 Agent 互相重复调用同一个 Tool，你会在协议层还是 runtime 层检测和去重？
- **L4 身份/前沿**：在 Agent Runtime 设计中，有没有可能用“单 Agent + 上下文隔离”来替代部分 Multi-Agent 场景？

#### 常见误区

- 认为 Agent 越多越好，把简单任务复杂化。

### Q23. 单 Agent 和 Multi-Agent 你怎么选？

#### 回答要点

- **定义/机制**：单 Agent 是模型在一个上下文中完成全部决策；Multi-Agent 是把任务拆给多个 Agent 协作。
- **为什么重要**：选择错误会导致上下文爆炸、延迟增加或调试困难。
- **工程做法**：默认先从单 Agent 做起；当上下文过大、角色冲突明显、工具边界不同、需要独立评估子任务时，再考虑 Multi-Agent。`长程 Agent 框架` 的设计哲学是：先验证单 Agent 是否能 hold 住，再决定拆分。
- **权衡/边界**：选择标准是收益是否大于复杂度；不要为了技术炫技而拆分。

#### 一句话版

默认单 Agent，只有当任务边界清晰、隔离收益明确时才拆成 Multi-Agent。

#### 追问点

- **L3 场景**：如果业务方要求“必须做 Multi-Agent” because 竞品都做了，你怎么说服或引导他们？
- **L4 身份/前沿**：在数据流水线 Agent 里，如果未来要支持多语言切片和本地化审核，你会选择 Multi-Agent 还是单 Agent + Skill？

#### 常见误区

- 把 Multi-Agent 当成先进性的标志，忽略它带来的复杂度和成本。

### Q24. 如果给 `长程 Agent 框架` 引入 目标团队 MLA 的 KV 压缩思想，你会怎么设计 Memory 层？

#### 回答要点

- **定义/机制**：MLA 通过低秩投影把 KV Cache 压缩到 latent vector，减少显存和访存；`长程 Agent 框架` 的 Memory 层也可以借鉴“只存压缩后的表示，按需解压/重构”。
- **为什么重要**：长程 Agent 的 Memory 如果全量存储原始上下文，读取和注入成本都会随任务长度增长；压缩后能支持更长任务、更低延迟。
- **工程做法**：在 `长程 Agent 框架` 中，可以把高频访问的上下文片段压缩成固定维度的 memory latent；读取时通过 projection 还原或直接用 latent 做相似性检索。需要保留原始内容的访问路径（key 引用），用于调试和关键决策。
- **权衡/边界**：压缩会损失部分细节，不能把所有 Memory 都压缩；关键决策点、用户明确指令应保留原始文本或摘要。

#### 一句话版

把 MLA 的 KV 压缩思想用到 `长程 Agent 框架` Memory，就是用低秩 latent 存储高频上下文，读取时按需解压或检索，同时保留关键原始内容。

#### 追问点

- **L3 场景**：如果压缩后的 Memory 检索出现语义漂移，你会怎么检测和修正？
- **L4 身份/前沿**：MLA 的压缩是模型推理时做的；`长程 Agent 框架` 的 Memory 压缩应该在写入时做还是读取时做？

#### 常见误区

- 认为所有 Memory 都可以压缩，忽视关键决策信息需要保真。

### Q25. 如果 Agent Runtime 要接入 GRPO 做在线任务后训练，Trace / Replay / Eval 要如何配合？

#### 回答要点

- **定义/机制**：GRPO 需要大量任务执行轨迹和奖励信号；Agent Runtime 的 Trace 提供轨迹、Replay 提供可复现的样本、Eval 提供奖励/质量判断。
- **为什么重要**：没有高质量、可复现、带奖励的轨迹，GRPO 很容易学偏或只能优化表面指标。
- **工程做法**：Trace 要记录完整决策链（模型输出、Tool 调用、上下文、结果）；Replay 要能按相同版本复现成功/失败案例；Eval 要给出多维奖励（任务成功、路径质量、成本、用户满意度）。`长程 Agent 框架` 中可以按任务阶段设检查点，把关键决策片段入库。
- **权衡/边界**：数据收集越全，存储和标注成本越高；需要在数据规模、标注质量和训练迭代速度之间取舍。

#### 一句话版

GRPO 需要 Agent Runtime 提供完整 Trace、可复现 Replay 和多维 Eval 奖励，三者共同构成后训练数据闭环。

#### 追问点

- **L3 场景**：如果 Eval 奖励稀疏（只有最终成功/失败），你会怎么设计中间步骤的奖励信号？
- **L4 身份/前沿**：在 `长程 Agent 框架` 中，哪些 Trace 片段最值得沉淀为 GRPO 训练样本？

#### 常见误区

- 只把最终成功当奖励，忽视路径质量和中间决策的可学习性。

### Q26. Mooncake / DualPipe 这类推理基础设施对 Agent Runtime 设计有什么启发？

#### 回答要点

- **定义/机制**：Mooncake 把 KV Cache  offload 到 CPU/SSD 做长上下文服务；DualPipe 做 Prefill-Decode 分离，让两种计算模式各自优化。它们都是推理基础设施层面的创新。
- **为什么重要**：Agent Runtime 处于模型之上、应用之下，必须理解这些基础设施的能力边界，才能设计合理的上下文调度策略。
- **工程做法**：Agent Runtime 可以：1）按任务长度和阶段选择模型服务节点；2）把长历史 Prefill 结果缓存复用，避免重复计算；3）对长程 Agent 做阶段化上下文注入，配合 PD 分离降低单请求延迟。
- **权衡/边界**：基础设施越复杂，Agent Runtime 的调度逻辑也越复杂；需要权衡延迟、成本、可靠性和实现复杂度。

#### 一句话版

Mooncake / DualPipe 启发 Agent Runtime 做分层上下文调度、KV Cache 复用和 PD 感知的任务编排。

#### 追问点

- **L3 场景**：如果 Agent Runtime 中一个长任务被拆成多次模型调用，如何利用 Mooncake 的 KV Cache 复用避免重复 Prefill？
- **L4 身份/前沿**：在 `长程 Agent 框架` 里，EventKey 和 Memory 设计如何与 Mooncake 的 KV Cache 管理形成互补？

#### 常见误区

- 把推理基础设施和 Agent Runtime 设计割裂，认为二者没有接口。
## 5. MCP / Tool 平台

### Q27. MCP 和普通函数调用相比，工程价值是什么？

#### 回答要点

- **定义/机制**：MCP 把工具和资源访问标准化为统一协议，宿主和 Tool 之间按统一 schema 交互。
- **为什么重要**：普通函数调用是代码级耦合，MCP 让能力可以跨宿主、跨语言、跨团队复用，并统一治理。
- **工程做法**：策略平台内容解析 Tool 以 MCP 形式提供后，多个 Agent 可以复用同一套解析能力，平台侧可以统一做参数校验、权限、缓存、限流、Trace 和版本管理。
- **权衡/边界**：MCP 增加了协议转换和治理成本；只有通用、边界清晰、需要多宿主复用的能力才值得 MCP 化。

#### 一句话版

MCP 的工程价值在于把通用能力标准化，实现跨宿主复用和统一治理，而不仅仅是一次函数调用。

#### 追问点

- **L3 场景**：如果 MCP Tool 的 schema 从 v1 升级到 v2，你如何保证旧 Agent 仍然能正常调用？
- **L4 身份/前沿**：对比 Function Call，你认为在 Agent Runtime 这种多 Agent 环境里，MCP 的核心价值是协议统一还是治理边界？

#### 常见误区

- 把 MCP 当成“远程函数调用”，忽略它在权限、缓存、观测、版本治理上的价值。

### Q28. 什么时候一个能力更适合做 MCP Tool？

#### 回答要点

- **定义/机制**：MCP Tool 适合通用、边界清晰、输入输出可结构化的能力。
- **为什么重要**：如果能力只被单一业务使用，直接写函数更简单；如果被多个 Agent/宿主复用，才需要 MCP 化。
- **工程做法**：策略平台内容解析 Tool 就是一个典型：图片/音频/视频/网页解析能力通用，需要被多个 Agent 调用，且需要独立做权限、缓存、限流和观测。
- **权衡/边界**：业务专属、变化快、语义不稳定的能力不适合 MCP 化，否则 schema 频繁变更会拖慢所有接入方。

#### 一句话版

当能力通用、边界清晰、需要多宿主复用和统一治理时，更适合做成 MCP Tool。

#### 追问点

- **L3 场景**：如果一个能力只有 2 个 Agent 用，但未来可能会扩展到 10 个，你会现在就 MCP 化，还是等需求明确？
- **L4 身份/前沿**：在 `长程 Agent 框架` 里，Skill 层和 MCP Tool 层哪个更适合先沉淀为可复用协议？

#### 常见误区

- 认为所有能力都应该 MCP 化，忽视协议维护和版本兼容成本。

### Q29. MCP 接入后还需要做哪些工程工作？

#### 回答要点

- **定义/机制**：MCP 只是协议层，接入后还要补齐工程保障。
- **为什么重要**：没有工程保障，MCP Tool 会在高并发、异常输入、权限越界等场景下出问题。
- **工程做法**：需要做参数校验和错误映射、权限和资源隔离、调用超时/重试/熔断、结果裁剪和缓存、Trace / 日志 / 指标。策略平台 MCP Tool 接入时就重点做了权限和缓存。
- **权衡/边界**：保障做得过重会影响延迟和接入成本；需要根据 Tool 的调用频率、副作用和重要性分层治理。

#### 一句话版

MCP 接入后还要做参数校验、权限隔离、超时重试熔断、结果裁剪、缓存和可观测。

#### 追问点

- **L3 场景**：如果一个 MCP Tool 下游 P99 延迟从 50ms 涨到 500ms，你会在 MCP 层做熔断，还是让调用 Agent 自己处理？
- **L4 身份/前沿**：在 Agent Runtime 里，Tool 的权限模型应该由 Agent runtime 统一管理，还是由每个 MCP Tool 自己管理？

#### 常见误区

- 认为 MCP 接入后就万事大吉，忽视参数校验、权限、限流和观测。

### Q30. 你如何评价 Tool 平台的好坏？

#### 回答要点

- **定义/机制**：Tool 平台的好坏取决于它能否让 Agent 稳定、高效、可治理地发现和使用工具。
- **为什么重要**：Tool 平台是 Agent Runtime 的核心基础设施，直接影响 Agent 的可靠性和开发效率。
- **工程做法**：评价维度包括：可发现性是否强、schema 是否清晰稳定、调试是否方便、可观测性是否足够、接入/治理/版本演进是否容易。`长程 Agent 框架` 中 Tool schema 的变更就需要版本化和灰度。
- **权衡/边界**：平台不能为了治理而牺牲灵活性；好的 Tool 平台应该让接入方在“严格 schema”和“快速迭代”之间找到平衡。

#### 一句话版

好的 Tool 平台要同时满足：易发现、schema 稳定、调试方便、可观测、易治理、易演进。

#### 追问点

- **L3 场景**：如果平台上某个 Tool 被 80% 的 Agent 调用，但它的 schema 设计有缺陷，你会选择兼容旧 schema 还是强制升级？
- **L4 身份/前沿**：如果 目标团队 开源了一套 Tool 平台标准，你会直接采用还是基于自己的 Agent Runtime 场景做定制？

#### 常见误区

- 只看 Tool 数量，不看 schema 质量、可观测性和治理成本。

## 6. Memory / Context Engineering

### Q31. Prompt、Context、Memory 的区别是什么？

#### 回答要点

- **定义/机制**：`Prompt` 是单次推理中的指令、角色、约束和格式要求；`Context` 是本次推理实际喂给模型的全部信息（Prompt + 历史 + Tool 结果 + 召回的 Memory）；`Memory` 是跨轮次、跨阶段持久化的结构化状态，包括用户偏好、任务进度、世界状态和工具结果摘要。
- **为什么重要**：三者混为一谈会导致“该控制的不控制、该持久化的不持久化”。Prompt 决定模型怎么答，Context 决定模型这次看到什么，Memory 决定系统长期记得什么。
- **工程做法**：在 `长程 Agent 框架` 中，Agent 之间传 `EventKey`（引用），真实内容由 runtime 从 Memory 装配成 Context，再和系统 Prompt 合并后进入模型；数据流水线 Agent 也会把视频理解结果按阶段写入 Memory，而非全量堆进 prompt。
- **权衡/边界**：过度拆分会增加 indirection 和读取延迟；过度统一则会让 Context 被历史噪声污染。

#### 一句话版

Prompt 是“怎么答”，Context 是“这次看到什么”，Memory 是“长期记得什么”；Memory 要经过筛选才能进入 Context。

#### 追问点

- **L3 场景**：如果一次任务里 Tool 返回了 5000 字原始结果，你会把它直接写进 Memory、直接进 Context，还是做摘要 + key 引用？
- **L4 身份/前沿**：目标团队 MLA 通过 KV 压缩降低长上下文成本，这种思想对 `长程 Agent 框架` 的 Memory 层有什么启发？

#### 常见误区

- 把聊天历史直接当 Memory，导致上下文膨胀、噪声干扰和错误信息固化。

### Q32. Memory 和 RAG 的区别是什么？

#### 回答要点

- **定义/机制**：`RAG` 是从外部知识库检索与当前查询相关的事实，补充到本次 Context；`Memory` 是系统对历史偏好、任务状态、工具结果和世界状态的持续维护。
- **为什么重要**：二者解决的问题不同。RAG 回答“我不知道但库里有”，Memory 回答“我之前知道/做过什么”。混用会让知识库被动态状态污染，或让记忆系统退化成检索系统。
- **工程做法**：RAG 通常用向量检索 + 重排，结果作为 Context 片段；Memory 在 `长程 Agent 框架` 中按 key 分层存储，读取时按任务阶段和类型筛选。必要时 RAG 结果也可以作为“外部知识型 Memory”进入长期记忆。
- **权衡/边界**：用户画像、产品手册这类静态内容适合 RAG；任务进度、最近操作、偏好设置这类动态状态适合 Memory。有重叠时要明确生命周期和更新责任人。

#### 一句话版

RAG 是从外部知识库查事实，Memory 是维护任务和用户的持续状态；二者可以互补，但不能互相替代。

#### 追问点

- **L3 场景**：如果用户问“我上周让你生成的报告呢”，这个信息应该从 RAG 还是 Memory 里取？为什么？
- **L4 身份/前沿**：在数据流水线 Agent 里，视频素材库的内容检索和用户的剪辑偏好，分别应该用 RAG 还是 Memory？

#### 常见误区

- 把 Memory 当成“小范围 RAG”，只关注向量相似度，忽略状态、生命周期和因果时序。

### Q33. 为什么不能把聊天历史全都当 Memory？

#### 回答要点

- **定义/机制**：聊天历史是原始交互日志，包含大量临时确认、错误尝试、口水话和中间状态；Memory 应该是经过抽象、清洗、验证后的高价值信息。
- **为什么重要**：直接堆历史会导致 Context 膨胀、噪声干扰、错误信息反复强化，甚至让模型在后续任务中重复之前的错误路径。
- **工程做法**：写入 Memory 前做摘要、提取关键事实、去重和置信度校验；区分 `working memory`（短期）和 `long-term memory`（长期）；对敏感操作保留来源 trace。
- **权衡/边界**：清理太激进会丢失用户个性化线索；太保守则污染长期记忆。需要基于信息价值和时效性做分层。

#### 一句话版

聊天历史是噪声很高的原始日志，Memory 必须是被抽象和筛选后的高价值信息，否则就是上下文污染。

#### 追问点

- **L3 场景**：如果用户在一次长对话中多次改需求，你会怎么防止“旧需求”被固化进 Memory 干扰后续任务？
- **L4 身份/前沿**：在 `长程 Agent 框架` 里，你会把 Agent 间通信的原始记录全写进 Memory，还是只写提取后的 `EventKey` 和状态摘要？

#### 常见误区

- 认为“记得越多越好”，忽视记忆的噪声、错误固化和检索命中率下降问题。

### Q34. 长期记忆系统最常见的问题是什么？

#### 回答要点

- **定义/机制**：长期记忆系统负责跨会话、跨任务的信息持久化，包括写入、检索、更新、清理四个环节，任一环节出问题都会变成“静默失败”。
- **为什么重要**：记忆问题不会立刻报错，但会长期拉低模型质量、用户体验和系统可信度。
- **工程做法**：常见问题包括：写入过多导致检索命中率低、过期信息未清理、错误信息被长期固化、语义相近但不相关的信息干扰、没有分层和生命周期管理、缺少来源 trace 和版本控制。
- **权衡/边界**：记忆系统需要在“记得全”和“记得准”之间取舍；过度追求召回率会引入噪声，过度追求精确度会丢失有价值的长期偏好。

#### 一句话版

长期记忆最危险的不是忘，而是记住错误、记住噪声、记住过期信息，还没人发现。

#### 追问点

- **L3 场景**：如果 Memory 检索结果中前 3 条都相关但互相矛盾，你会怎么设计冲突消解策略？
- **L4 身份/前沿**：如果 `长程 Agent 框架` 要支持跨会话的长期任务，你会怎么设计 Memory 的 TTL、版本和回滚机制？

#### 常见误区

- 只关注“能不能写进去”，不关注“能不能读得准”“能不能清得掉”。

### Q35. 你如何设计 Memory 的写入策略？

#### 回答要点

- **定义/机制**：写入策略决定什么信息、在什么时机、以什么形式进入记忆系统，是记忆质量的第一道关卡。
- **为什么重要**：写入过多会污染记忆、降低检索质量；写入过少会丢失关键状态。写入时机和格式也影响后续读取效率。
- **工程做法**：在 `长程 Agent 框架` 中，优先写入稳定、高价值、跨阶段有用的信息；按类型分层（用户偏好、任务状态、工具结果摘要、世界状态）；明确生命周期（session 级、短期、长期）；写入前做清洗、去重、摘要和置信度校验；关键写入保留来源和版本。
- **权衡/边界**：实时写入能保证最新状态，但可能写入噪声；批量摘要能减少写入量，但会丢失细粒度信息。自动写入效率高，但关键信息可能需要规则或人工校验。

#### 一句话版

写入策略的核心是：只写稳定、高价值、跨阶段有用的信息，并且写之前要清洗、摘要、分层。

#### 追问点

- **L3 场景**：如果 Agent 在一次任务中产生了 20 条中间状态，你会全部写入 Memory，还是只写最终状态和关键决策点？
- **L4 身份/前沿**：在数据流水线 Agent 中，视频理解结果、用户剪辑指令、生成进度，分别应该按什么频率和格式写入 Memory？

#### 常见误区

- 认为每轮交互都要写 Memory，导致记忆系统被低价值信息淹没。

### Q36. 你如何设计 Memory 的读取策略？

#### 回答要点

- **定义/机制**：读取策略决定在当前任务阶段从记忆中召回哪些信息、以什么顺序、以多大粒度进入 Context。
- **为什么重要**：读错和读多都会让模型决策变差；读取策略直接影响延迟、成本和回答质量。
- **工程做法**：先判断当前任务阶段需要什么信息；按阶段和类型筛选，不做全量注入；优先注入当前决策真正依赖的最小集合；使用 `EventKey` 引用替代原文注入；结合相关性、时效性、重要性和用户偏好排序。
- **权衡/边界**：召回过多会引入噪声和增加 token 成本；召回过少会遗漏关键信息。复杂检索能提升质量，但会增加读取延迟。

#### 一句话版

读取策略的核心是：按阶段、按类型、按最小必要集合召回，优先用 key 引用而不是原文注入。

#### 追问点

- **L3 场景**：如果一次任务需要同时引用“用户上周的偏好”和“刚刚产生的工具结果”，你会如何平衡二者的召回权重？
- **L4 身份/前沿**：如果 目标团队 的 MLA 能把 KV Cache 压缩到很小，是否意味着 `长程 Agent 框架` 可以放松对 Memory 读取的裁剪策略？

#### 常见误区

- 每次读取都把长期记忆全量注入 prompt，既不经济也容易让模型注意力分散。

### Q37. 你如何判断哪些上下文应该进 prompt？

#### 回答要点

- **定义/机制**：进入 prompt 的信息必须满足“能改变当前决策”且“收益大于成本”。Context Engineering 就是做这个取舍的工程实践。
- **为什么重要**：Prompt 空间是有限的注意力资源；塞入无关信息会稀释关键信号、增加成本和延迟。
- **工程做法**：用四问法则判断：当前任务目标是否需要它？它是否能改变当前决策？它比 token 成本更有价值？它是否会引入歧义、冲突或噪声？在 `长程 Agent 框架` 中，还会结合任务阶段和 `EventKey` 引用做最小化注入。
- **权衡/边界**：保守注入可能缺信息导致模型失败；激进注入可能污染注意力。需要在具体任务上做 A/B 和 Eval。

#### 一句话版

只有能改变当前决策、收益大于成本且不引入噪声的信息，才应该进入 prompt。

#### 追问点

- **L3 场景**：如果系统 prompt 里已经有一段“永远遵守的安全策略”，而用户当前请求疑似冲突，你会把这段策略和用户请求一起放进 prompt，还是单独做 guardrail？
- **L4 身份/前沿**：在 Agent Runtime 系统中，当上下文窗口越来越大，Context Engineering 的价值会上升还是下降？为什么？

#### 常见误区

- 认为“上下文越多越好”，把历史、工具结果、相关知识全塞进去。

### Q38. 为什么说 Context Engineering 是 Agent Runtime 的核心？

#### 回答要点

- **定义/机制**：Context Engineering 是围绕“给模型看什么、什么时候给、给多少、怎么组织”的一整套工程实践，包括信息组织、阶段注入、长度控制和冲突消解。
- **为什么重要**：模型能力上限不仅取决于模型本身，还取决于 Context；Agent Runtime 的稳定性、成本和质量都取决于 Context Engineering 是否到位。
- **工程做法**：阶段化注入（不同任务阶段给不同信息）、Context Budget 管理、Memory 分层、冲突消解机制、Tool 结果摘要、A/B 测试和 Eval 闭环。`长程 Agent 框架` 的 `EventKey` 和阶段化 Memory 读取就是 Context Engineering 的具体落地。
- **权衡/边界**：过度工程会增加延迟、维护复杂度和 rigidity；不足则模型表现不稳定。需要在可控性和灵活性之间找平衡。

#### 一句话版

Context Engineering 是 Agent Runtime 的核心，因为它决定模型能不能在正确的时机看到正确数量的正确信息。

#### 追问点

- **L3 场景**：如果一个 Agent Runtime 任务越跑越慢，你会优先检查模型、Tool，还是 Context 注入策略？
- **L4 身份/前沿**：目标团队 的长上下文和 KV 压缩技术，会如何改变 Agent Runtime 中 Context Engineering 的设计重点？

#### 常见误区

- 把 Context Engineering 简单等同于“写 prompt”，忽略信息组织、阶段注入和冲突消解。

### Q39. 你项目里 memory key / event key 的设计价值是什么？

#### 回答要点

- **定义/机制**：`EventKey` / `MemoryKey` 是一种轻量级引用，把大对象和模型 Context 解耦，真实内容存储在 Memory 中，需要时由 runtime 按需装配。
- **为什么重要**：长程 Agent 中上下文很容易爆炸；用 key 引用可以控制 Context 大小、支持审计、权限隔离、缓存和重放，也让 Agent 间通信更轻量。
- **工程做法**：在 `长程 Agent 框架` 中，Agent 之间不传大段上下文而传 `EventKey`，runtime 按任务阶段和权限装配真实内容；key 设计包含命名空间、版本、TTL 和来源 trace。数据流水线 Agent 也可以用 key 引用视频特征、剪辑片段等大对象。
- **权衡/边界**：好处是控制成本和增强可控性；代价是引入 indirection，key 失效或读取延迟会带来新问题，需要兜底策略。

#### 一句话版

EventKey 把“上下文引用”和“上下文内容”解耦，让大对象管理回到 runtime，从而控制成本并支持审计和重放。

#### 追问点

- **L3 场景**：如果一个 `EventKey` 引用的内容在任务中途被清理，Agent 行为会出什么问题？你会怎么设计兜底？
- **L4 身份/前沿**：如果 目标团队 MLA 的 KV 压缩思想应用到 `长程 Agent 框架` 的 Memory，你会如何重新设计 key 的存储和读取策略？

#### 常见误区

- 只把 key 当成“省 token 的手段”，忽略它在状态管理、审计、权限和重放上的价值。

## 7. Trace / Eval / Replay

### Q40. Trace 和日志有什么区别？

#### 回答要点

- **定义/机制**：日志是离散的事件记录；Trace 是围绕一次任务或请求，把跨阶段、跨服务的调用、状态变化和关键上下文关联起来的完整链路。
- **为什么重要**：Agent 系统中单点日志看不出问题，必须看模型输入输出、Tool 调用、Memory 读写、重试和异常的完整链路才能定位根因。
- **工程做法**：Trace 需要包含用户输入、系统 Prompt、当前 Context 切片、模型请求/响应、Tool 调用参数与结果、Memory 读写 key、重试/异常/终止原因、最终输出。`长程 Agent 框架` 中 Trace 还要把 `EventKey` 的装配过程也记录下来。
- **权衡/边界**：Trace 越全越好调试，但存储和采样成本越高；需要按任务重要性和失败率做采样和保留策略。

#### 一句话版

日志记录离散事件，Trace 把一次任务全链路上的模型、Tool、Memory 和异常串起来。

#### 追问点

- **L3 场景**：如果一次 Agent 任务失败，Trace 显示 Tool 调用成功、模型输出也合理，你会怎么继续定位？
- **L4 身份/前沿**：在 `长程 Agent 框架` 里，如果 Agent 间只传 `EventKey`，Trace 要如何记录才能还原真实 Context 装配过程？

#### 常见误区

- 只记录模型输入输出，不记录 Tool 调用、Memory 读写和 Context 变化，导致无法定位中间步骤问题。

### Q41. Agent Trace 至少应该记录什么？

#### 回答要点

- **定义/机制**：Agent Trace 是 Agent 运行时的可观测核心，需要覆盖“模型看到了什么、想了什么、调了什么、结果如何、状态怎么变”。
- **为什么重要**：没有完整 Trace，Agent 就是黑盒；无法做失败归因、Replay、Eval 和持续优化。
- **工程做法**：至少记录：用户原始输入、系统 Prompt 版本、当前 Context 切片、模型请求和响应、Tool 调用参数与返回、Memory 读写 key 与摘要、重试/异常/终止原因、最终输出和耗时/cost。数据流水线 Agent 还要记录多模态理解结果和剪辑决策。
- **权衡/边界**：记录过全会影响性能和存储；需要在“可调试性”和“成本”之间做采样和字段裁剪。

#### 一句话版

Agent Trace 至少要记录：输入、Prompt、Context、模型输出、Tool 调用、Memory 读写、异常和最终输出。

#### 追问点

- **L3 场景**：如果 Trace 数据量太大导致采样率不得不降到 1%，你会优先保留哪些字段？
- **L4 身份/前沿**：`长程 Agent 框架` 的 `EventKey` 设计下，Trace 应该记录 key 还是记录 key 展开后的真实内容？

#### 常见误区

- 只记录最终输出和错误码，忽略中间步骤和 Context 变化。

### Q42. 为什么 Replay 很重要？

#### 回答要点

- **定义/机制**：Replay 是在受控环境下重放一次 Agent 任务完整执行过程的能力，包括输入、模型响应、Tool 调用和状态变化。
- **为什么重要**：复杂 Agent 系统的失败往往是多因素耦合的，Replay 让问题可复现、可对比、可回归测试，是调试和优化的基础。
- **工程做法**：Replay 需要捕获完整的 Trace、外部依赖的 mock 或快照、Prompt/Tool/Memory 的版本。`长程 Agent 框架` 中可以通过 `EventKey` 和 Memory 快照实现重放；数据流水线 Agent 中则需要 mock 多模态理解结果。
- **权衡/边界**：完全 Replay 需要 mock 所有外部依赖，成本高；可以分层做：本地单步 Replay、沙箱 Replay、线上影子 Replay。

#### 一句话版

Replay 让 Agent 任务可复现、可对比、可回归，是调试复杂系统的关键能力。

#### 追问点

- **L3 场景**：如果一个 Agent 任务依赖外部 API 返回实时数据，你如何设计 Replay 才能让结果稳定？
- **L4 身份/前沿**：在 Agent Runtime 系统中，Replay 能力对模型后训练（如 GRPO）的数据采集有什么价值？

#### 常见误区

- 认为 Replay 只是“重新跑一遍”，忽略外部依赖 mock、状态快照和版本一致性问题。

### Q43. Eval 应该评什么？

#### 回答要点

- **定义/机制**：Agent Eval 是系统化的质量评估，需要从任务完成度、路径质量、成本效率、用户体验多个维度度量。
- **为什么重要**：单一指标（如任务成功率）会掩盖路径冗余、成本高、体验差等问题；Agent Runtime 需要多维度权衡。
- **工程做法**：核心指标包括：任务成功率、工具成功率、平均步数、重试率、延迟、token 成本、用户满意度 proxy、人工标注一致率。`长程 Agent 框架` 中还会评 `EventKey` 装配准确率和 Memory 命中率。
- **权衡/边界**：指标过多会导致指标疲劳和优化目标冲突；需要分层：北极星指标 + 诊断指标 + guardrail 指标。

#### 一句话版

Eval 要评任务成功率、工具成功率、路径效率、成本、延迟和用户体验，不能只看最终输出。

#### 追问点

- **L3 场景**：如果任务成功率提升但平均步数和 token 成本大幅上涨，你会怎么判断这次迭代是否值得上线？
- **L4 身份/前沿**：在 Agent Runtime 团队，Eval 指标应该如何驱动 Prompt、Tool 和模型的迭代优先级？

#### 常见误区

- 只看任务最终是否正确，忽略路径质量、成本和中间失败。

### Q44. 没有标准答案的 Agent 任务怎么做 Eval？

#### 回答要点

- **定义/机制**：开放式 Agent 任务需要用多维指标和分层评估替代单一正确率，包括过程指标和结果指标。
- **为什么重要**：很多 Agent Runtime 任务没有唯一答案，必须用结构化检查、规则检查、模型评判、人工抽检和用户反馈共同评估。
- **工程做法**：分层评估：第一层任务完成（是否达成用户目标）、第二层路径质量（步骤是否冗余、是否稳定）、第三层成本效率（token/延迟/调用次数）。结合规则检查、LLM-as-judge、人工抽检和用户反馈。`长程 Agent 框架` 中对长程任务会按阶段设检查点。
- **权衡/边界**：LLM-as-judge 成本低但可能带偏；人工抽检准但贵；需要在不同迭代阶段选用不同评估方式。

#### 一句话版

没有标准答案的 Agent 任务要用多维指标 + 规则 + LLM judge + 人工抽检 + 用户反馈做分层评估。

#### 追问点

- **L3 场景**：如果 LLM-as-judge 的打分和人类标注一致性只有 60%，你会怎么改进评估流程？
- **L4 身份/前沿**：在数据流水线 Agent 中，如何设计 Eval 来度量“生成内容是否符合用户风格”这种主观目标？

#### 常见误区

- 因为没有标准答案就放弃系统评估，只靠“看起来对”或人工逐条看。

### Q45. 如何定位是模型问题、工具问题还是上下文问题？

#### 回答要点

- **定义/机制**：三类问题有各自的诊断信号：模型问题表现为即使 Context 正确、Tool 结果正确，模型仍做不出合理决策；工具问题表现为调用参数或返回结果异常；上下文问题表现为关键信息缺失、噪声干扰或冲突。
- **为什么重要**：Agent Runtime 中很多问题被误判为模型问题，实则是 Context 设计或 Tool 设计问题；定位错误会导致无效迭代。
- **工程做法**：用控制变量法：去掉 Tool 看模型是否能合理规划；固定 Context 看 Tool 调用是否正确；用 Replay 对比不同 Context 下的模型输出。`长程 Agent 框架` 中通过 `EventKey` 展开可以单独验证 Context 装配是否正确。
- **权衡/边界**：真实场景中往往是多因素耦合，需要分层归因而不是简单归类。

#### 一句话版

定位三类问题要用控制变量：去掉 Tool 看模型、固定 Context 看 Tool、用 Replay 对比不同 Context。

#### 追问点

- **L3 场景**：如果模型输出和 Tool 结果都看起来正确，但最终任务失败，你会怎么继续归因？
- **L4 身份/前沿**：在 Agent Runtime 团队，你会如何设计一个自动化的归因工具，把失败快速归类到模型/Tool/Context？

#### 常见误区

- 只看最终失败，不分析中间步骤；一出问题就归咎于模型能力不足。

## 8. LLM 与推理基础

### Q46. Transformer 的基本组成是什么？

#### 回答要点

- **定义/机制**：Transformer 由 token embedding、positional encoding、多头 self-attention、前馈网络（FFN）、残差连接、层归一化组成，通过多层堆叠形成深层上下文表示。
- **为什么重要**：它是现代 LLM 的架构基础，Attention 机制让模型能捕获长距离依赖，也是 KV Cache、长上下文、推理成本的源头。
- **工程做法**：实际系统中关心的是：embedding 维度、层数、注意力头数、FFN 中间维度，它们共同决定模型大小、显存占用和推理延迟。Agent 调用时要意识到每多一层、每多一个 token 都会放大成本。
- **权衡/边界**：Transformer 的优势是并行训练和长距离建模；代价是计算和显存随序列长度增长，长上下文始终是瓶颈。

#### 一句话版

Transformer = embedding + self-attention + FFN + 残差/归一化，多层堆叠形成上下文表示，也是推理成本的源头。

#### 追问点

- **L3 场景**：在 Agent 多轮 loop 中，如果每次调用都带很长的历史，Transformer 的哪些特性会让成本和延迟快速放大？
- **L4 身份/前沿**：目标团队 MLA 对 Transformer 的 KV Cache 做了压缩，这对 `长程 Agent 框架` 这种长程 Agent 的上下文管理有什么启发？

#### 常见误区

- 只记模块名称，不理解它们如何影响成本、延迟和显存。

### Q47. Attention 的直觉是什么？

#### 回答要点

- **定义/机制**：Attention 让当前 token 在生成表示时，不只看自己，而是对历史 token 做加权聚合；权重由 query 和 key 的相似度决定，value 携带实际语义。
- **为什么重要**：它是 Transformer 捕获上下文关系的核心，也是 KV Cache 存在的原因：decode 时只需要算新 token 的 Q，复用历史 K/V。
- **工程做法**：多头注意力把表示空间分成多个子空间，分别学习不同模式；在实际推理中，QK^T 的显存和计算是优化重点，flash attention 等算法就是围绕这一点。
- **权衡/边界**：Attention 让模型看到全部历史，但代价是序列变长时计算和显存增加；稀疏注意力、线性注意力、MQA/GQA/MLA 都是在 trade-off 质量和效率。

#### 一句话版

Attention 的直觉是：当前 token 生成表示时，按相关性对历史 token 做加权聚合。

#### 追问点

- **L3 场景**：如果 Attention 权重集中在错误的历史 token 上，可能是 Context 设计问题还是模型问题？
- **L4 身份/前沿**：目标团队 MLA 把 KV Cache 压缩到一维 latent vector，你怎么看这种压缩对长程 Agent 的利弊？

#### 常见误区

- 只理解为“加权平均”，忽略 Q/K/V 分工、多头机制和 KV Cache 的工程影响。

### Q48. KV Cache 的作用是什么？

#### 回答要点

- **定义/机制**：KV Cache 在 decode 阶段缓存历史 token 的 key 和 value，避免每生成一个新 token 都重新计算全部历史的 K/V。
- **为什么重要**：没有 KV Cache，decode 复杂度会随序列长度爆炸；有了它，decode 每次只需处理新 token 的 attention，显著降低延迟和计算量。
- **工程做法**：KV Cache 通常以张量形式驻留显存，大小与层数、头数、序列长度、头维度、batch 成正比。vLLM 的 PagedAttention、目标团队 MLA 都是在优化 KV Cache 的存储和访问效率。
- **权衡/边界**：KV Cache 加速推理但占用显存；上下文越长、batch 越大，占用越大，是长上下文和并发推理的主要瓶颈。

#### 一句话版

KV Cache 把历史 token 的 K/V 存起来复用，让 decode 不用每次重算全部历史，是推理加速的关键。

#### 追问点

- **L3 场景**：如果一个 Agent 任务跑了 20 轮，每轮都保留完整历史，KV Cache 占用会如何变化？你会怎么优化？
- **L4 身份/前沿**：目标团队 MLA 把 KV Cache 压缩后，是否意味着 Agent 可以放松对上下文长度的控制？

#### 常见误区

- 只知道 KV Cache 加速，不清楚它如何占用显存、如何成为长上下文瓶颈。

### Q49. Prefill 和 Decode 有什么区别？

#### 回答要点

- **定义/机制**：Prefill 是把输入 prompt 整段编码，计算所有 token 的 KV Cache；Decode 是逐个生成新 token，复用 KV Cache 并计算新 token 的 attention。
- **为什么重要**：二者特性完全不同：Prefill 计算密集、吞吐导向、可并行；Decode 访存密集、延迟敏感、串行性强。Agent 系统中一次请求通常包含多次 Prefill+Decode 交替。
- **工程做法**：在线服务要平衡 prefill 和 decode 的调度；长 prompt 会拉长首 token 时间（TTFT），长 decode 会拉高每个 token 延迟。vLLM、SGLang 等调度器会针对二者做 chunked prefill、decode-priority 等优化。
- **权衡/边界**：Prefill 阶段可以大开 batch 提升吞吐；Decode 阶段 batch 太大会增加尾延迟。Agent 多轮调用中需要分别优化。

#### 一句话版

Prefill 是整段编码进 KV Cache，计算密集；Decode 是逐 token 生成，延迟敏感。

#### 追问点

- **L3 场景**：Agent 每轮调用都产生一次新的 prefill，这对 TTFT 和整体延迟有什么影响？
- **L4 身份/前沿**：目标团队 的 PD 分离（Prefill-Decode disaggregation）对 Agent Runtime 这种多轮 Agent 系统有什么价值？

#### 常见误区

- 把 Prefill 和 Decode 当成同一个阶段优化，忽略它们在计算模式、延迟目标和调度策略上的差异。

### Q50. 长上下文为什么昂贵？

#### 回答要点

- **定义/机制**：长上下文昂贵来自三个方面：Prefill 阶段要处理更多 token；KV Cache 占用显著增加；Decode 每一步都要访问更长历史。
- **为什么重要**：Agent 多轮 loop 中上下文天然容易膨胀，长上下文成本会被多次调用放大。
- **工程做法**：优化方向包括：更严格的 Context Engineering、上下文摘要、KV Cache 压缩（MLA、量化、稀疏化）、PagedAttention、PD 分离、按阶段注入信息。`长程 Agent 框架` 的 `EventKey` 本质也是从工程侧控制上下文长度。
- **权衡/边界**：长上下文提升模型理解能力，但成本和延迟也上升；不是所有任务都需要全历史，关键是“给对信息”而不是“给全信息”。

#### 一句话版

长上下文贵在三处：Prefill token 多、KV Cache 大、Decode 每步访问历史长。

#### 追问点

- **L3 场景**：如果一个 Agent 任务因为上下文太长导致 OOM，你会优先砍历史、压缩 KV，还是拆分任务？
- **L4 身份/前沿**：目标团队 把上下文窗口扩展到百万级，Agent Runtime 是否还需要做上下文裁剪？

#### 常见误区

- 认为窗口变大就能无限堆历史，忽视成本、延迟和注意力稀释。

### Q51. 显存主要消耗在哪？

#### 回答要点

- **定义/机制**：推理时显存主要由四部分构成：模型权重、KV Cache、中间激活/运行时缓冲、以及 batch/并发带来的额外占用。
- **为什么重要**：显存通常是推理的第一瓶颈，尤其 Agent 多轮、长上下文、高并发场景下，KV Cache 会快速膨胀。
- **工程做法**：估算显存时要分别算：权重（和参数量、精度有关）、KV Cache（和层数/头数/序列长/batch 有关）、激活（和 batch/隐藏维度有关）。优化手段包括量化、减少 batch、KV Cache 压缩、上下文裁剪、offloading。
- **权衡/边界**：降低显存往往以精度、延迟或吞吐为代价；需要根据在线/离线、延迟敏感程度做选择。

#### 一句话版

显存主要吃在模型权重、KV Cache、中间激活和 batch/并发开销上。

#### 追问点

- **L3 场景**：如果一个 70B 模型在 batch=1、4K 上下文时 OOM，你会先怀疑权重还是 KV Cache？
- **L4 身份/前沿**：目标团队 MLA 把 KV Cache 压得很小，这对显存分配策略和并发上限有什么影响？

#### 常见误区

- 只看模型大小，忽视 KV Cache 和 batch 对显存的动态占用。

### Q52. 显存不够时你会考虑哪些方向？

#### 回答要点

- **定义/机制**：显存不够时可以从“减输入、减模型、减并发、优化存储”四个方向入手。
- **为什么重要**：不同方向对应不同 trade-off，不能一刀切；需要结合任务延迟、质量和成本目标选择。
- **工程做法**：具体方向包括：更严格的 context engineering（缩短/摘要历史）、降低 batch 或并发、换更小模型或分层路由（简单任务用小模型）、量化（INT8/INT4/FP8）、KV Cache 压缩/分页（PagedAttention、MLA）、工具结果外置存储、分布式推理。
- **权衡/边界**：砍上下文可能影响质量；降并发影响吞吐；量化影响精度；需要按场景分层治理。

#### 一句话版

显存不够时可以从减输入、减模型、减并发、优化 KV Cache 存储四个方向入手。

#### 追问点

- **L3 场景**：如果 Agent 高峰时显存被打满，你会优先限流、降 batch，还是动态切换小模型？
- **L4 身份/前沿**：在 `长程 Agent 框架` 里，是否可以通过 `EventKey` 把部分上下文从 GPU 显存转移到外部存储来缓解压力？

#### 常见误区

- 只想着买卡或扩显存，不从算法和工程侧做显存优化。

### Q53. 为什么 Agent 特别需要关注推理成本？

#### 回答要点

- **定义/机制**：Agent 不是单次调用，而是多轮 loop，每一轮都可能有 prefill+decode，还会调用 Tool 并把结果回填，成本和延迟会被多次放大。
- **为什么重要**：如果没有成本控制，复杂任务可能一次调用就消耗大量 token，导致产品不可持续。
- **工程做法**：控制成本的手段包括：上下文裁剪、Memory 分层、Tool 结果摘要、按任务难度路由不同模型、限制最大步数、缓存常见结果。`长程 Agent 框架` 的 `EventKey` 和阶段性 Memory 注入就是典型的成本控制设计。
- **权衡/边界**：过度控制成本会牺牲任务成功率；需要在质量、延迟、成本之间找到产品可接受的平衡。

#### 一句话版

Agent 是多轮 loop，每次调用和工具结果回填都会放大 token 和延迟，所以成本必须纳入设计。

#### 追问点

- **L3 场景**：如果一个 Agent 任务平均 10 步，每步 2K token，你会怎么设定成本预算和告警？
- **L4 身份/前沿**：目标团队 推理模型能力强但成本高，Agent Runtime 是否应该按任务难度做模型路由？

#### 常见误区

- 只关注单次调用成本，忽视多轮 loop 和上下文膨胀带来的累计成本。

### Q54. 采样参数通常影响什么？

#### 回答要点

- **定义/机制**：采样参数（temperature、top-p/top-k、repetition penalty 等）控制模型输出的随机性和多样性。temperature 平滑/锐化概率分布，top-p 限制候选集合。
- **为什么重要**：Agent 中不同步骤对稳定性的要求不同：工具调用需要高确定性，头脑风暴需要一定创造性，采样参数直接影响成功率和可重复性。
- **工程做法**：结构化输出/Tool 调用通常用低 temperature + 限制 top-p；候选生成/探索阶段可以适当放开。关键路径建议固定 seed 或做多次采样投票。
- **权衡/边界**：温度太低会重复和缺乏创造性；太高会不稳定、容易偏离 schema。没有一组参数适合所有任务。

#### 一句话版

采样参数控制输出的随机性和多样性；Agent 中工具调用要低温度，探索阶段可适当放开。

#### 追问点

- **L3 场景**：如果模型频繁以不同格式调用同一个 Tool，你会调整采样参数还是改 schema/Prompt？
- **L4 身份/前沿**：目标团队 强推理模型对采样参数是否比普通模型更不敏感？Agent Runtime 中是否需要重新调参？

#### 常见误区

- 一套采样参数走天下，不区分任务阶段对稳定性的要求。

### Q55. 函数调用 / 工具调用和普通文本生成的差异是什么？

#### 回答要点

- **定义/机制**：普通文本生成关注自然语言质量；函数调用/工具调用要求模型输出符合预定义 schema 的结构化参数，并驱动工程系统执行。
- **为什么重要**：Tool 调用是 Agent 与外部世界交互的桥梁，参数错误会直接导致执行失败或副作用；需要把模型输出纳入类型校验、错误恢复和重试链路。
- **工程做法**：Tool 调用需要清晰的 schema、参数校验、错误映射、结果裁剪、幂等保护和可观测。`长程 Agent 框架` 中 Tool 返回会经过结构化处理再进入下一轮 Context；数据流水线 Agent 中工具调用直接影响视频处理链路。
- **权衡/边界**：过度限制 schema 会降低模型灵活性；太灵活又容易参数幻觉。需要在稳定性和通用性之间迭代。

#### 一句话版

工具调用不是生成自然语言，而是输出符合 schema 的结构化参数并进入工程执行链路。

#### 追问点

- **L3 场景**：如果 Tool 调用频繁出现参数缺失或类型错误，你会优先改 schema、改 Prompt，还是加工程校验？
- **L4 身份/前沿**：目标团队 强推理模型在工具调用上是否更可靠？Agent Runtime 是否可以放松 schema 约束？

#### 常见误区

- 把工具调用当成普通文本生成评估，只看语言质量不看参数正确性和执行结果。

### Q56. vLLM 的 PagedAttention 如何解决 KV Cache 内存碎片问题？

#### 回答要点

- **定义/机制**：PagedAttention 把 KV Cache 划分成固定大小的 block（类似 OS 虚拟内存分页），按需分配和共享，而不是给每个序列预分配连续大内存。
- **为什么重要**：传统 KV Cache 分配方式会产生大量内部/外部碎片，限制 batch 大小和并发；PagedAttention 提高显存利用率，支持更大 batch 和更长上下文。
- **工程做法**：在 `长程 Agent 框架` 或 Agent Runtime 中调用 vLLM 时，可以受益于这种内存管理；设计上下文时要意识到 KV Cache 是 block 化的，可以共享前缀（Prefix Caching）。
- **权衡/边界**：分页增加内存管理复杂度；block 大小需要调优，太大浪费，太小增加管理开销。

#### 一句话版

PagedAttention 把 KV Cache 分 block 管理，按需分配和共享，减少碎片、提升显存利用率和并发。

#### 追问点

- **L3 场景**：如果 Agent 多轮调用有很多共享系统 prompt，vLLM 的 Prefix Caching 能带来什么收益？
- **L4 身份/前沿**：在 Agent Runtime 中，是否可以把共享的 system prompt / 工具说明做成常驻 Prefix Cache？

#### 常见误区

- 只看 PagedAttention 提升吞吐，不理解它解决的是内存碎片和利用率问题。

### Q57. Prefix Caching 在 Agent 多轮调用中有什么价值？

#### 回答要点

- **定义/机制**：Prefix Caching 缓存已计算过的 KV Cache（通常是系统 prompt、工具说明、固定上下文），新请求如果有相同前缀可以直接复用。
- **为什么重要**：Agent 多轮调用中，系统 prompt 和工具 schema 往往不变；Prefix Caching 能显著降低 TTFT 和计算成本。
- **工程做法**：Agent Runtime 可以把稳定的系统 prompt、角色定义、工具说明组织成可缓存前缀；把动态的用户输入和工具结果放在后缀。`长程 Agent 框架` 的 Context Engineering 要有 prefix/suffix 意识。
- **权衡/边界**：Prefix 太长会占用显存；如果 prefix 经常变化，缓存命中率会下降。

#### 一句话版

Prefix Caching 让 Agent 多轮调用中稳定的系统 prompt 和工具说明只算一次，降低延迟和成本。

#### 追问点

- **L3 场景**：如果一次 Agent 任务中系统 prompt 在不同步骤略有不同，你还会用 Prefix Caching 吗？
- **L4 身份/前沿**：在 `长程 Agent 框架` 的阶段性 Memory 注入中，如何设计才能让更多内容命中 Prefix Cache？

#### 常见误区

- 认为 Prefix Caching 只用于完全相同的多请求，忽视部分匹配和 prefix 设计。
## 9. Multi-Agent / Planning

### Q58. Planning 一定要交给模型吗？

#### 回答要点

- **定义/机制**：Planning 可以交给模型（由 LLM 自主分解步骤）、交给工程（预定义 Workflow/DAG）、或做混合（工程骨架 + 模型填充细节）。
- **为什么重要**：Agent Runtime 的稳定性要求不能把复杂任务完全交给模型的不确定性；但过度工程化又会限制模型处理开放任务的能力。
- **工程做法**：在 `长程 Agent 框架` 中，常见模式是“工程定义阶段目标和检查点，模型负责阶段内的具体决策”；数据流水线 Agent 中“下载 → 转码 → 投稿”是工程固定链路，而“内容理解 → 切片点选择”是模型决策。
- **权衡/边界**：流程约束强、错误代价高、可验证的场景适合工程规划；目标开放、路径不确定、需要语义理解时适合模型规划。

#### 一句话版

Planning 不一定交给模型；Agent Runtime 中通常采用“工程定骨架、模型填细节”的混合模式。

#### 追问点

- **L3 场景**：如果一个任务 80% 步骤固定、20% 需要模型判断，你会做成 Workflow + Skill 混合，还是让模型端到端规划？
- **L4 身份/前沿**：目标团队 强推理模型规划能力更强，Agent Runtime 是否可以把更多规划权交给模型？边界在哪里？

#### 常见误区

- 认为所有任务都应该让模型自由规划，忽视工程约束对稳定性和可观测性的价值。

### Q59. 什么时候该让工程约束流程，什么时候让模型自由规划？

#### 回答要点

- **定义/机制**：工程约束适合“确定性高、可验证、错误代价高、需要稳定复现”的部分；模型规划适合“目标开放、路径不确定、需要语义理解、变化快”的部分。
- **为什么重要**：错误分配会导致该稳定的环节失控，或该灵活的地方被卡死。
- **工程做法**：经验法则是“确定性归工程，不确定性归模型”。`长程 Agent 框架` 中 Skill 和 Workflow 就是这一思想的体现：Workflow 处理固定流程，Skill 封装半结构化任务，模型在 Skill 内做具体决策。
- **权衡/边界**：工程约束过强会让系统僵化；模型规划过强则难以调试和保证稳定性。需要清晰的阶段检查点和回滚机制。

#### 一句话版

已知流程、错误代价高的部分交给工程；目标开放、路径不确定的部分交给模型。

#### 追问点

- **L3 场景**：在数据流水线 Agent 中，“视频下载”和“切片文案生成”分别该由工程约束还是模型规划？
- **L4 身份/前沿**：如果 目标团队 推理模型能稳定输出高质量规划，Agent Runtime 的 Workflow 层是否会逐渐变薄？

#### 常见误区

- 把“模型自由”当成先进性标志，忽视工程约束在关键路径上的必要性。

### Q60. Agent 规划失败时常见原因有哪些？

#### 回答要点

- **定义/机制**：Agent 规划失败是指模型生成的执行计划无法达成用户目标，或执行过程中偏离目标。
- **为什么重要**：规划失败是 Agent Runtime 最常见的失败模式之一，且往往被误认为是模型能力不足，实则是目标、上下文、工具或约束设计问题。
- **工程做法**：常见原因包括：目标理解错误、工具能力边界不清、上下文缺失或污染、中间结果未结构化、缺少阶段性校验和纠偏、奖励/反馈信号设计不当。`长程 Agent 框架` 中通过阶段检查点、Trace 和 Replay 来定位具体原因。
- **权衡/边界**：完全防止规划失败不现实，关键是快速检测、可回滚、可重放，并把失败案例沉淀为 Eval 数据集。

#### 一句话版

规划失败通常不是模型不会，而是目标不清、工具边界不清、上下文不对或缺少阶段校验。

#### 追问点

- **L3 场景**：如果 Agent 在执行中越走越偏，你会设计哪些“护栏”让它提前终止或回退？
- **L4 身份/前沿**：在 `长程 Agent 框架` 中，你如何利用 Replay 来系统性分析规划失败的根因？

#### 常见误区

- 规划失败就归结为模型能力不行，而不检查目标、工具、上下文和约束设计。

### Q61. 你如何看待 Subagent？

#### 回答要点

- **定义/机制**：Subagent 是把任务拆分成更小的、由独立 Agent 负责的子任务，通过协议协作完成。它本质上是任务分解 + 上下文隔离的工程手段。
- **为什么重要**：当单 Agent 上下文过大、角色冲突明显、工具边界不同或子任务需要独立评估时，Subagent 可以降低复杂度和成本。
- **工程做法**：在 `长程 Agent 框架` 中，Subagent 拆分会同时考虑上下文隔离、工具边界和独立 Trace；数据流水线 Agent 的“内容理解”和“切片生成”理论上可以拆给不同 Subagent。关键是拆分时要有明确的边界、协议和 Eval。
- **权衡/边界**：Subagent 增加了通信、协议和调试成本；默认先从单 Agent 做起，除非任务分解天然成立且收益明确。

#### 一句话版

Subagent 适合任务边界清晰、需要上下文隔离和独立评估的场景；不要为了拆分而拆分。

#### 追问点

- **L3 场景**：如果两个 Subagent 同时读写同一个 Memory，你会怎么设计并发控制和冲突消解？
- **L4 身份/前沿**：在 Agent Runtime 设计中，有没有可能用“单 Agent + 上下文隔离”替代部分 Subagent 场景？

#### 常见误区

- 为了 Multi-Agent/Subagent 而拆分，增加不必要的通信成本和调试难度。

## 10. 规则引擎 / DAG Runtime / DSL

### Q62. 规则引擎最重要的设计点是什么？

#### 回答要点

- **定义/机制**：规则引擎是把业务策略从代码中抽离出来，通过声明式规则做匹配、执行和冲突消解的系统。
- **为什么重要**：它让策略变更不需要改代码，支持快速迭代、灰度、A/B 测试；但如果设计不好，会变成难以理解和治理的黑盒。
- **工程做法**：最重要的设计点包括：表达能力与可控性的平衡、执行效率、冲突处理和优先级机制、规则的校验/灰度/回滚、可观测和可解释性。策略平台 / 规则引擎就重点做了配置热发布、灰度和可观测。
- **权衡/边界**：表达能力越强，越接近通用编程语言，治理难度越大；需要在灵活性和可验证性之间取舍。

#### 一句话版

规则引擎最重要的是平衡表达能力与可控性，并做好冲突处理、灰度、回滚和可观测。

#### 追问点

- **L3 场景**：如果两条规则优先级相同但结论冲突，你会怎么设计默认策略？
- **L4 身份/前沿**：规则引擎的“策略配置化”经验，如何迁移到 Agent Runtime 的 Prompt/Policy 配置化系统？

#### 常见误区

- 只追求表达能力强，忽视校验、灰度、审批、回滚和可观测。

### Q63. DSL 为什么容易失控？

#### 回答要点

- **定义/机制**：DSL 是为特定领域设计的专用语言，表达能力一旦过强，就会接近通用编程语言，带来理解和治理成本。
- **为什么重要**：DSL 失控会导致规则难以阅读、测试、审计，新手不敢改、老手改错不知道。
- **工程做法**：控制 DSL 的方法包括：限制语法子集、强制类型和静态检查、提供可视化编辑器、做版本管理和审批流、设置灰度和回滚机制、限制递归和复杂控制流。
- **权衡/边界**：DSL 太弱无法满足业务需求，太强难以治理；通常要在“能表达业务”和“能被非程序员理解”之间找平衡。

#### 一句话版

DSL 失控是因为表达力过强接近通用语言，导致难读、难测、难治理。

#### 追问点

- **L3 场景**：如果业务方坚持要在 DSL 里加循环和递归，你会怎么拒绝或限制？
- **L4 身份/前沿**：在 Agent Runtime 的 Prompt/Policy 配置化里，如何防止配置变成“另一种失控的 DSL”？

#### 常见误区

- 为了灵活性把 DSL 做成图灵完备，结果失去可验证性和可维护性。

### Q64. DAG Runtime 和 Agent Runtime 的异同是什么？

#### 回答要点

- **定义/机制**：DAG Runtime 执行预定义的有向无环图，节点是确定性任务；Agent Runtime 执行由模型动态决策的任务链，路径开放。
- **为什么重要**：二者处理的问题域不同，但都需要依赖管理、状态维护、执行调度、可观测和失败恢复。
- **工程做法**：相同点：都要处理依赖、状态、执行、可观测、失败恢复。不同点：DAG 路径确定，Agent Runtime 路径开放、依赖模型输出，因此更需要 trace、guardrail、上下文管理和反馈闭环。`长程 Agent 框架` 的 runtime 就是在借鉴 DAG Runtime 的工程稳定性的同时，保留模型决策的灵活性。
- **权衡/边界**：DAG 适合确定性流程，Agent Runtime 适合开放任务；Hybrid 架构是 Agent Runtime 的常见选择。

#### 一句话版

DAG Runtime 执行预定义的确定性流程，Agent Runtime 执行模型动态决策的开放路径；二者都需要状态、调度、观测和恢复。

#### 追问点

- **L3 场景**：如果一个 Agent 任务中某些子步骤必须严格按顺序执行且不能重排，你会用 DAG 约束还是让模型自己规划？
- **L4 身份/前沿**：在 Agent Runtime 中，是否可以把“模型决策出的步骤”动态编译成临时 DAG 执行？

#### 常见误区

- 把 Agent Runtime 简单等同于 DAG Runtime，忽视模型决策带来的不确定性和上下文管理需求。

### Q65. 为什么 SDK 本地预解析配置能提效？

#### 回答要点

- **定义/机制**：SDK 本地预解析配置是指在启动期或服务初始化时，把远程配置反序列化、校验并转换成内存中的结构化对象，避免运行时反复解析。
- **为什么重要**：规则引擎/DSL 配置如果被高频访问，运行时解析会成为热点路径瓶颈；预解析能显著提升吞吐、降低延迟。
- **工程做法**：在 SDK 启动时拉取配置，做语法/语义校验、构建索引、生成内存结构；运行时直接读内存对象。配合配置版本号和热更新机制，可以在变更时重新预解析并切换。
- **权衡/边界**：预解析增加启动时间和内存占用；配置更新时需要重新加载，切换过程要保证一致性。

#### 一句话版

SDK 本地预解析配置能把运行时的解析开销移到启动期，提升热点路径性能并提前发现配置错误。

#### 追问点

- **L3 场景**：如果配置体积很大，启动期预解析变慢，你会怎么优化加载速度？
- **L4 身份/前沿**：在 Agent Runtime 的 Prompt/Policy 配置化中，是否也需要类似的“预编译”或“缓存”机制？

#### 常见误区

- 只关注预解析的性能收益，忽视配置更新一致性、启动时间和内存占用。

### Q66. 推荐系统 Infra 中的 DSL/算子编排和规则引擎有什么异同？

#### 回答要点

- **定义/机制**：推荐 Infra DSL 用于描述特征、模型、后处理等算子的编排；规则引擎 DSL 用于描述策略决策和条件执行。二者都是领域专用语言，但面向的问题域不同。
- **为什么重要**：理解异同有助于把规则引擎经验迁移到推荐系统或其他 AI Infra 场景，也是 Agent Runtime 中 Prompt/Policy 配置化的基础。
- **工程做法**：相同点：都需要做表达力控制、版本管理、灰度回滚、可观测。不同点：推荐 DSL 更偏数据流和算子依赖，规则引擎更偏条件判断和策略冲突消解。
- **权衡/边界**：不能把推荐 DSL 直接当规则引擎用，也不能把规则引擎做得像通用数据流系统；要按问题域选择合适的抽象。

#### 一句话版

推荐 DSL 偏算子/数据流编排，规则引擎 DSL 偏条件/策略决策；二者都需要控制表达力、做版本和灰度。

#### 追问点

- **L3 场景**：如果推荐系统 DSL 中某个算子失败，你会让整条链路失败还是走降级路径？
- **L4 身份/前沿**：在 Agent Runtime 的 Prompt/Policy 配置化中，更偏向推荐 DSL 还是规则引擎 DSL 的抽象？

#### 常见误区

- 把两种 DSL 混为一谈，忽视数据流和决策逻辑的本质差异。

### Q67. 如果设计一个支持特征工程、模型推理、后处理算子的 DSL，你会如何控制其表达能力？

#### 回答要点

- **定义/机制**：这种 DSL 需要让业务方描述“从特征到最终输出”的完整链路，但又不能让它变成无法验证的通用编程语言。
- **为什么重要**：表达能力过强会导致校验难、测试难、运行时风险高；过弱又无法满足业务需求。
- **工程做法**：控制方法包括：限定算子集合（白名单）、强制声明输入输出 schema、静态类型检查、禁止递归和任意循环、提供可视化编排、版本化和审批流、沙箱执行高危算子。
- **权衡/边界**：控制越严格，业务自由度越低；需要在安全性和灵活性之间分层治理。

#### 一句话版

用白名单算子、强制 schema、静态检查、禁止递归、版本审批和沙箱执行来控制 DSL 表达能力。

#### 追问点

- **L3 场景**：如果业务方坚持要在 DSL 里写自定义 Python UDF，你会怎么平衡安全和灵活？
- **L4 身份/前沿**：在 Agent Runtime 的 Tool/Skill 编排中，是否也需要类似的“算子白名单”机制？

#### 常见误区

- 为了灵活性把 DSL 做成图灵完备，导致难以验证和治理。
### Q68. 配置热更新最大的风险是什么？

#### 回答要点

- **定义/机制**：配置热更新是在不重启服务的情况下动态加载新配置，风险在于错误配置会直接进入线上、新旧版本行为不一致、配置与代码契约失配。
- **为什么重要**：在 Agent Runtime/Agent 系统中，Prompt、Policy、Tool schema 都可能通过配置管理；热更新失败会直接影响模型行为，且难以快速定位。
- **工程做法**：防控措施包括：配置校验和沙箱测试、灰度发布、版本号管理、快速回滚、配置与代码契约检查、可观测和告警。策略平台 / 规则引擎的配置热发布就做了这些。
- **权衡/边界**：热更新提升迭代速度，但增加风险；关键配置更新需要审批和灰度，非关键配置可以自动推送。

#### 一句话版

配置热更新最大的风险是错误配置直接上线且缺少灰度、回滚和校验。

#### 追问点

- **L3 场景**：如果一个配置热更新后导致 1% 的用户请求失败，你会怎么快速回滚和定位？
- **L4 身份/前沿**：在 Agent Runtime 中，Prompt 和 Policy 的配置热更新应该如何做版本控制和 A/B 测试？

#### 常见误区

- 只追求热更新速度，忽视灰度、审批、回滚和配置契约校验。

## 11. 后端与系统设计

### Q69. 如何保护一个被 Agent 高频调用的 Tool 服务？

#### 回答要点

- **定义/机制**：Tool 服务被 Agent 高频调用时，需要防止被打垮、被滥用、被错误重试导致级联故障。
- **为什么重要**：Agent 多轮 loop 会放大调用频率，一旦 Tool 服务不稳定，整个 Agent 任务都会失败。
- **工程做法**：保护措施包括：超时/重试/熔断/限流、场景级和内容级缓存、区分只读和副作用工具、幂等保护、降级路径、trace 和指标监控。`长程 Agent 框架` 中 Tool 层会统一做这些治理。
- **权衡/边界**：保护过强会影响 Agent 成功率和延迟；需要根据 Tool 的重要性、副作用和调用模式分层治理。

#### 一句话版

保护 Tool 服务要做超时、限流、熔断、缓存、幂等、降级，并按 Tool 副作用分层治理。

#### 追问点

- **L3 场景**：如果下游服务被 Agent 高频调用打垮，你的降级策略是什么？是返回空结果、返回缓存、还是让 Agent 终止任务？
- **L4 身份/前沿**：在 Agent Runtime 的 Tool 平台中，这些保护措施应该由 Agent runtime 统一提供，还是由每个 Tool 自己实现？

#### 常见误区

- 只讲架构不量化，没有具体的限流阈值、超时策略和降级方案。

### Q70. 为什么重试有时候很危险？

#### 回答要点

- **定义/机制**：重试是在调用失败后再次尝试，但对非幂等操作或已高负载的下游，重试会放大问题。
- **为什么重要**：Agent 多轮 loop 中 Tool 调用失败很常见，盲目重试会导致重复扣款、重复提交、下游雪崩。
- **工程做法**：安全重试需要：幂等设计、退避策略（指数退避）、重试预算、区分可重试错误和不可重试错误、熔断保护。`长程 Agent 框架` 中 Tool 层会统一封装重试策略。
- **权衡/边界**：完全不重试会降低成功率；无限制重试会放大故障。需要按错误类型和下游状态动态调整。

#### 一句话版

重试危险是因为非幂等操作会被重复执行，且高负载时盲目重试会放大故障。

#### 追问点

- **L3 场景**：如果一个非幂等 Tool 因为超时返回失败，但实际已经执行成功，你会怎么设计重试策略避免重复执行？
- **L4 身份/前沿**：在 Agent Runtime 中，重试策略应该由 Agent runtime 统一管理，还是让模型自己决定是否重试？

#### 常见误区

- 认为所有失败都可以重试，忽视幂等性、退避和重试预算。

### Q71. 幂等如何设计？

#### 回答要点

- **定义/机制**：幂等是指同样的请求执行一次和执行多次效果相同。对 Agent 调用的 Tool 来说，幂等是安全重试和故障恢复的基础。
- **为什么重要**：Agent 多轮 loop 中网络超时、重试、异常终止很常见；没有幂等保护，重复调用会导致数据错误和副作用。
- **工程做法**：使用唯一请求 ID 或业务去重键；服务端保存执行状态或结果；重试时先查询是否已执行；明确哪些操作天然不幂等，需要额外保护（如加锁、状态机）。`长程 Agent 框架` 中 Tool schema 可以标记幂等性，runtime 据此决定是否重试。
- **权衡/边界**：幂等设计会增加存储和查询开销；不是所有操作都能做到完全幂等，需要状态机和补偿机制。

#### 一句话版

幂等设计用唯一 ID + 状态保存 + 去重查询，让同样的请求执行多次效果相同。

#### 追问点

- **L3 场景**：如果 Tool 调用方和下游系统都实现了幂等，但两个下游系统之间的消息顺序无法保证，你会怎么保证最终一致性？
- **L4 身份/前沿**：在 Agent Runtime 的 Tool 平台中，幂等性是否应该作为 Tool schema 的必填属性？

#### 常见误区

- 认为只要加了唯一 ID 就幂等，不保存状态、不做去重查询。

### Q72. 你怎么理解最终一致性？

#### 回答要点

- **定义/机制**：最终一致性是指在分布式系统中，短时间内不同副本或不同链路可能不一致，但在没有新写入后会收敛到一致状态。
- **为什么重要**：Agent/Tool 调用往往涉及多个服务，强一致性代价高；最终一致性是更常见的工程选择，但需要定义可接受的不一致窗口。
- **工程做法**：关键是定义一致窗口、设计补偿机制（重试、对账、回滚）、保证幂等、使用异步消息或状态机同步。`长程 Agent 框架` 中跨 Agent 的状态更新可以通过事件日志 + 补偿达到最终一致。
- **权衡/边界**：最终一致性降低延迟和复杂度，但增加了理解和调试难度；金融、库存等场景可能需要强一致性。

#### 一句话版

最终一致性允许短暂不一致，但要在无新写入后收敛，并设计好补偿和对账机制。

#### 追问点

- **L3 场景**：如果一个 Agent 任务调用了多个 Tool，部分成功部分失败，你会怎么设计补偿机制？
- **L4 身份/前沿**：在 Agent Runtime 中，模型决策和工具执行的一致性边界应该如何设计？

#### 常见误区

- 把最终一致性当成“暂时不一致没关系”，不定义窗口、不做补偿。

### Q73. 限流、降级、熔断分别解决什么问题？

#### 回答要点

- **定义/机制**：限流控制流入系统的请求速率，防止被压垮；降级在资源不足时保留核心功能，牺牲非核心功能；熔断在下游异常时快速失败，防止级联故障。
- **为什么重要**：Agent 高频调用 Tool 时，这三者是稳定性保障的基本手段。
- **工程做法**：限流可以按 QPS/并发/用户维度；降级可以返回默认值、缓存或简化结果；熔断可以基于错误率/延迟。`长程 Agent 框架` 中 Tool runtime 可以统一提供这些能力，并按 Tool 重要性配置不同策略。
- **权衡/边界**：限流太严影响成功率；降级过度影响用户体验；熔断过早会误伤；都需要根据业务容忍度调整。

#### 一句话版

限流防压垮、降级保核心、熔断防级联；三者结合构成 Tool 服务的稳定性防线。

#### 追问点

- **L3 场景**：如果 Agent 高峰期大量调用某 Tool，限流和熔断同时触发，你会怎么决定是让 Agent 重试还是走降级？
- **L4 身份/前沿**：在 Agent Runtime 的 Tool 平台中，限流/降级/熔断策略应该由平台统一配置，还是交给每个 Agent 自己决定？

#### 常见误区

- 只讲概念不量化，没有具体的阈值、触发条件和恢复策略。

## 12. Go / C++ / 语言基础

### Q74. goroutine 和线程是什么关系？

#### 回答要点

- **定义/机制**：goroutine 是 Go runtime 管理的用户态轻量级协程，多个 goroutine 复用少量操作系统线程，由 runtime 调度器（G-M-P 模型）进行调度。
- **为什么重要**：goroutine 创建和切换成本远低于线程，适合高并发服务；`长程 Agent 框架` 作为 Go 实现的 Agent runtime，依赖 goroutine 处理大量并发 Agent 任务和 Tool 调用。
- **工程做法**：通过 `go` 关键字启动；使用 channel 做同步和通信；使用 `sync` 包处理共享状态；配合 `context.Context` 做取消和超时控制。
- **权衡/边界**：goroutine 轻量但不免费，大量 goroutine 仍会带来调度开销和内存占用；共享状态仍需正确处理数据竞争。

#### 一句话版

goroutine 是 Go runtime 调度的用户态协程，复用线程、切换成本低，适合高并发但也要注意数据竞争。

#### 追问点

- **L3 场景**：如果 `长程 Agent 框架` 里同时跑了上万个 goroutine，你会关注哪些调度或内存指标？
- **L4 身份/前沿**：在 Agent runtime 中，goroutine 的并发模型和 Python 的 asyncio 相比，各自适合什么场景？

#### 常见误区

- 认为 goroutine 完全没有成本，忽视数据竞争、channel 阻塞和 goroutine 泄漏。

### Q75. `context.Context` 在 Go 中有什么用？

#### 回答要点

- **定义/机制**：`context.Context` 是 Go 中传递取消信号、超时/截止时间、请求级元数据的标准机制，通过函数参数逐级传递。
- **为什么重要**：在 `长程 Agent 框架` 这种长程 Agent 系统中，一个用户请求可能触发多个 goroutine 和多次 Tool 调用；需要统一的取消和超时控制，避免资源泄漏。
- **工程做法**：用 `context.WithCancel`/`WithTimeout`/`WithDeadline` 创建；用 `ctx.Done()` 监听取消；把 trace ID、用户 ID 等元数据放 `context.WithValue`。注意不要把业务对象塞进 context。
- **权衡/边界**：context 传太深会让 API 签名冗长；传太浅又控制不到子任务。需要按调用边界合理使用。

#### 一句话版

`context.Context` 用于传递取消信号、超时和请求级元数据，是 Go 并发控制的基础设施。

#### 追问点

- **L3 场景**：如果一个 Tool 调用超时，但 Agent 还想继续尝试其他方案，你会怎么设计 context 的取消范围？
- **L4 身份/前沿**：在 `长程 Agent 框架` 的多 Agent 协作中，context 应该如何跨 Agent 传递取消信号？

#### 常见误区

- 把 context 当成“大包袱”存任意业务对象，或忽视取消信号导致 goroutine 泄漏。

### Q76. 什么是 Go 中的逃逸分析？

#### 回答要点

- **定义/机制**：逃逸分析是 Go 编译器判断变量应该在栈上分配还是逃逸到堆上的过程。如果变量生命周期超出函数作用域或被引用传递，通常会逃逸到堆。
- **为什么重要**：栈分配由函数返回自动回收，堆分配增加 GC 压力；理解逃逸分析有助于写出内存友好的代码。
- **工程做法**：通过 `go build -gcflags='-m'` 查看逃逸分析结果；减少不必要的指针返回、闭包捕获和接口装箱。`长程 Agent 框架` 中高频路径的内存分配需要关注逃逸情况。
- **权衡/边界**：不是所有逃逸都是坏事，有时候为了正确性必须逃逸；关键是避免高频路径上的意外堆分配。

#### 一句话版

逃逸分析决定变量分配在栈还是堆；栈分配高效，堆分配增加 GC 压力。

#### 追问点

- **L3 场景**：如果你发现 `长程 Agent 框架` 的热点函数有很多变量逃逸到堆，你会怎么优化？
- **L4 身份/前沿**：在 Agent runtime 这种需要低延迟的场景下，逃逸分析对你的编码习惯有什么影响？

#### 常见误区

- 认为所有逃逸都要避免，或完全不理解为什么某些变量会逃逸到堆。

### Q77. 虚函数调用在什么时候决定？

#### 回答要点

- **定义/机制**：虚函数调用是动态绑定，具体调用哪个实现由运行时对象的实际类型决定；编译期只生成通过虚表（vtable）查找的代码。
- **为什么重要**：虚函数是多态的基础，但动态绑定带来一定的运行时开销；在性能敏感路径（如推理服务）需要关注虚函数调用的成本。
- **工程做法**：C++ 中通过 `virtual` 关键字和继承实现；调用时通过对象指针/引用触发动态绑定。理解虚表布局有助于分析性能和内存布局。
- **权衡/边界**：虚函数提供灵活性但失去内联和静态优化机会；在极端性能场景可能需要用模板或函数指针替代。

#### 一句话版

虚函数调用在运行时通过虚表决定，支持多态但有一定开销。

#### 追问点

- **L3 场景**：如果一个 C++ 推理服务的热点路径频繁调用虚函数，你会怎么优化？
- **L4 身份/前沿**：`长程 Agent 框架` 选择 Go 而不是 C++ 实现 runtime，虚函数/接口调用的性能差异是否是考虑因素之一？

#### 常见误区

- 认为虚函数调用和普通函数调用性能一样，忽视动态绑定和虚表查找开销。

### Q78. `vector` 扩容为什么会导致迭代器失效？

#### 回答要点

- **定义/机制**：`std::vector` 底层是连续内存，当容量不足时扩容需要重新分配更大的内存块，并把元素搬移到新内存。
- **为什么重要**：扩容后原先指向旧内存的迭代器、指针、引用都会失效，继续使用会导致未定义行为。
- **工程做法**：如果预知道大小，用 `reserve` 预分配；需要稳定引用的场景考虑用 `list`/`deque` 或存索引而不是存指针。在推理服务中处理张量或缓存时要注意这一点。
- **权衡/边界**：`vector` 随机访问快、缓存友好，但扩容和插入有失效风险；选择容器要权衡访问模式和稳定性需求。

#### 一句话版

`vector` 扩容会重新分配内存并搬移元素，导致指向旧内存的迭代器和引用失效。

#### 追问点

- **L3 场景**：如果你需要在遍历 `vector` 时安全删除元素，应该怎么做？
- **L4 身份/前沿**：在 C++ 推理服务中，有哪些场景会因为这个特性导致隐蔽 bug？

#### 常见误区

- 不知道 `push_back` 可能触发扩容，拿着旧迭代器继续用。

### Q79. `map` 和 `unordered_map` 有什么差别？

#### 回答要点

- **定义/机制**：`std::map` 基于红黑树实现，键有序，操作复杂度 `O(log n)`；`std::unordered_map` 基于哈希表，键无序，平均 `O(1)`、最坏 `O(n)`。
- **为什么重要**：选择取决于是否需要有序遍历、对最坏情况延迟的容忍度、以及键类型的哈希支持。
- **工程做法**：需要范围查询或有序遍历时用 `map`；需要平均快速查找且不关心顺序时用 `unordered_map`。在性能敏感场景要注意 `unordered_map` 的内存开销和哈希冲突。
- **权衡/边界**：`map` 稳定但慢；`unordered_map` 快但最坏情况不可控、内存占用大。C++20 的 `std::unordered_map` 有一些改进，但仍有权衡。

#### 一句话版

`map` 是有序树，复杂度 `O(log n)`；`unordered_map` 是哈希表，平均 `O(1)` 但无序且最坏情况差。

#### 追问点

- **L3 场景**：如果你要实现一个需要频繁插入和查找的缓存，你会选 `map` 还是 `unordered_map`？
- **L4 身份/前沿**：在 `长程 Agent 框架` 的 Memory 索引设计中，如果要在 C++ 里实现，你会怎么选择容器？

#### 常见误区

- 只看平均复杂度，不看最坏情况、内存开销和是否需要有序性。

## 13. 操作系统 / 网络 / SQL / 前端

### Q80. 进程、线程、协程的区别是什么？

#### 回答要点

- **定义/机制**：进程是资源分配和隔离的基本单位，拥有独立地址空间；线程是 CPU 调度的基本单位，共享进程资源；协程是用户态调度的轻量执行单元，由 runtime 或程序自身管理切换。
- **为什么重要**：三者的切换成本、隔离性和并发模型不同，决定了它们在 Agent/推理系统中的适用场景。
- **工程做法**：进程适合强隔离（如多租户、推理服务沙箱）；线程适合利用多核并行；协程适合高并发 IO 密集型任务。`长程 Agent 框架` 用 goroutine（协程）处理大量并发 Agent 任务。
- **权衡/边界**：进程隔离好但切换和通信成本高；线程共享内存但同步复杂；协程轻量但调度不由 OS 保证公平性。

#### 一句话版

进程重隔离，线程重调度，协程轻量用户态切换；三者按隔离、并行、并发需求选用。

#### 追问点

- **L3 场景**：如果 `长程 Agent 框架` 要运行不可信代码或第三方 Tool，你会用进程隔离还是容器隔离？
- **L4 身份/前沿**：在推理服务中，Prefill 和 Decode 是否适合用不同线程/进程模型调度？

#### 常见误区

- 死记硬背定义，不会结合场景分析切换成本、隔离性和同步复杂度。

### Q81. 死锁的四个必要条件是什么？

#### 回答要点

- **定义/机制**：死锁发生的四个必要条件是：互斥、占有且等待、不可剥夺、循环等待。同时满足时才可能死锁。
- **为什么重要**：Agent 系统中多个 goroutine/线程可能竞争 Memory、Tool 调用、状态锁；理解死锁有助于设计避免策略。
- **工程做法**：预防方法包括：破坏循环等待（按固定顺序加锁）、减少持有锁的时间、使用超时锁、避免嵌套锁、使用 channel 替代共享内存（Go 风格）。
- **权衡/边界**：过度防御会增加代码复杂度；通常采用“避免 + 检测 + 超时”组合策略。

#### 一句话版

死锁需要互斥、占有且等待、不可剥夺、循环等待四个条件同时满足；破坏任一条件可预防。

#### 追问点

- **L3 场景**：在 `长程 Agent 框架` 中，如果两个 Agent 同时想更新同一个 Memory，你会怎么设计加锁顺序避免死锁？
- **L4 身份/前沿**：在分布式 Agent 系统中，死锁检测和超时机制应该如何设计？

#### 常见误区

- 只背四个条件，不会结合实际系统设计预防策略。

### Q82. 共享内存的原理是什么？

#### 回答要点

- **定义/机制**：共享内存让多个进程把同一段物理内存映射到各自虚拟地址空间，实现高速数据交换，无需内核态拷贝。
- **为什么重要**：在推理服务中，共享内存常用于 Host-GPU、多进程推理 worker、IPC 等场景，能显著降低数据传输开销。
- **工程做法**：需要配合信号量、互斥锁、原子操作等同步机制使用。`长程 Agent 框架` 中如果涉及多进程共享状态，也可以考虑共享内存 + 锁，但更多场景用 channel 或网络通信。
- **权衡/边界**：共享内存效率高但同步复杂，容易出竞态；调试和跨机器部署也比消息传递困难。

#### 一句话版

共享内存让多个进程映射同一段物理内存，避免拷贝，但需要额外同步机制。

#### 追问点

- **L3 场景**：如果一个推理服务用共享内存把模型输入传给 GPU，你会怎么处理同步和一致性问题？
- **L4 身份/前沿**：在 `长程 Agent 框架` 中，什么情况下你会选择共享内存而不是网络/RPC 做进程间通信？

#### 常见误区

- 只关注共享内存的高性能，忽视同步复杂性和一致性风险。

### Q83. SQL 的执行顺序是什么？

#### 回答要点

- **定义/机制**：SQL 逻辑执行顺序是：`FROM` → `JOIN` → `WHERE` → `GROUP BY` → `HAVING` → `SELECT` → `DISTINCT` → `ORDER BY` → `LIMIT`。
- **为什么重要**：理解执行顺序才能写出正确、高效的 SQL，尤其在涉及聚合、子查询和别名时。
- **工程做法**：写 SQL 时要让过滤条件尽量下沉到 WHERE，减少后续处理数据量；避免在 WHERE 中对字段做函数运算导致索引失效；复杂查询用 EXPLAIN 分析执行计划。
- **权衡/边界**：逻辑顺序和物理执行顺序可能不同，优化器会重写；但写查询时仍要按逻辑顺序理解语义。

#### 一句话版

SQL 逻辑执行顺序：FROM/JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT。

#### 追问点

- **L3 场景**：如果一条 SQL 在 `WHERE` 中对索引字段用了函数，导致全表扫描，你会怎么改写？
- **L4 身份/前沿**：在 Agent 的元数据/日志查询系统中，SQL 执行顺序的理解对索引设计有什么影响？

#### 常见误区

- 按书写顺序理解 SQL，不知道 SELECT 在 HAVING 之后执行，导致别名和聚合用错。

### Q84. HTTP2 相比 HTTP1.1 的主要改进是什么？

#### 回答要点

- **定义/机制**：HTTP/2 在 HTTP/1.1 语义兼容的基础上，引入二进制分帧、多路复用、头部压缩（HPACK）、服务端推送等机制。
- **为什么重要**：HTTP/1.1 的队头阻塞和大量连接开销在高并发场景下成为瓶颈；HTTP/2 能显著提升 Web 服务性能。
- **工程做法**：多路复用让一条连接上并行传输多个请求/响应；头部压缩减少重复 header 开销；二进制分帧提高解析效率。但 HTTP/2 仍有 TCP 层队头阻塞问题，HTTP/3 用 QUIC 解决。
- **权衡/边界**：HTTP/2 提升高并发性能，但实现更复杂、调试更困难；某些场景（如大量小请求）收益不明显。

#### 一句话版

HTTP/2 通过二进制分帧、多路复用、头部压缩减少连接数和队头阻塞。

#### 追问点

- **L3 场景**：如果 Agent 服务大量调用外部 HTTP API，HTTP/2 的多路复用能带来什么收益？
- **L4 身份/前沿**：HTTP/3 的 QUIC 对低延迟 Agent Tool 调用有什么潜在价值？

#### 常见误区

- 认为 HTTP/2 能解决所有延迟问题，忽视 TCP 队头阻塞和应用层优化。

### Q85. 为什么会跨域？哪些场景需要跨域处理？

#### 回答要点

- **定义/机制**：跨域是浏览器的同源策略限制：协议、域名、端口任一不同即视为跨域。它是浏览器安全模型的一部分，不是服务端天然限制。
- **为什么重要**：前后端分离、多域名、CDN、第三方资源等场景都会遇到跨域；处理不当会导致前端请求被拦截或产生安全漏洞。
- **工程做法**：常见处理方式：CORS（服务端配置 Access-Control-Allow-Origin）、JSONP（已不推荐）、反向代理、postMessage。生产环境要限制允许的 origin，避免 `*` 滥用。
- **权衡/边界**：CORS 配置太宽松有安全风险；太严格会影响业务。需要按最小权限原则配置。

#### 一句话版

跨域是浏览器同源策略限制，常见于前后端分离和多域资源访问，通常通过 CORS 或反向代理解决。

#### 追问点

- **L3 场景**：如果一个 Agent 的前端控制台要调用不同域的 API，你会选择 CORS 还是网关统一代理？
- **L4 身份/前沿**：在 Agent Runtime 的开发者平台中，跨域配置和安全策略应该如何设计？

#### 常见误区

- 认为跨域是服务端错误，或为了方便把 CORS 配置成 `*` 允许所有来源。

### Q86. CSS 优先级怎么比较？

#### 回答要点

- **定义/机制**：CSS 优先级由选择器特性和来源共同决定。`!important` 优先级最高；行内样式高于内部/外部样式；选择器按 ID、class/属性/伪类、标签/伪元素的数量比较；相同特性后定义生效。
- **为什么重要**：前端基础题，考察对样式冲突和调试的理解。
- **工程做法**：实际项目中优先用 class 和语义化命名，避免滥用 `!important` 和行内样式；复杂项目用 CSS 模块或原子化 CSS 降低优先级冲突。
- **权衡/边界**：优先级机制灵活但容易失控；规范化和工程化比记忆优先级更重要。

#### 一句话版

CSS 优先级：`!important` > 行内 > ID > class/属性/伪类 > 标签/伪元素；相同特性后定义生效。

#### 追问点

- **L3 场景**：如果项目里出现大量 `!important` 导致样式难以维护，你会怎么重构？
- **L4 身份/前沿**：在 Agent 的开发者平台或可视化编排界面中，CSS 工程化对可维护性有什么影响？

#### 常见误区

- 只靠 `!important` 解决优先级问题，导致样式越来越难以维护。

### Q87. 浏览器为什么有跨域限制，但后端服务之间没有这个问题？

#### 回答要点

- **定义/机制**：跨域限制是浏览器安全模型的一部分，防止网页脚本随意访问其他源的资源；后端服务之间调用不经过浏览器，不受同源策略约束。
- **为什么重要**：理解这个区别能正确设计前后端交互和微服务通信。后端服务虽然不受同源策略限制，但仍需自己做认证、授权和 TLS。
- **工程做法**：浏览器端用 CORS 或反向代理；后端服务之间用 API Key、JWT、mTLS、IP 白名单等手段做安全控制。
- **权衡/边界**：同源策略保护用户但不限制服务端；服务端安全不能依赖浏览器同源策略，必须自己做零信任设计。

#### 一句话版

跨域是浏览器安全策略，后端服务之间不受此限制，但后端必须自己做认证和权限控制。

#### 追问点

- **L3 场景**：如果前端通过浏览器直接调用第三方 LLM API，会遇到哪些安全和跨域问题？
- **L4 身份/前沿**：在 Agent Runtime 的 Tool 平台中，如何区分“浏览器端 Tool 调用”和“服务端 Tool 调用”的安全模型？

#### 常见误区

- 认为后端服务之间没有跨域就天然安全，忽视认证和权限控制。

## 14. GPU / 推理工程

### Q88. 为什么 GPU 适合做大模型推理？

#### 回答要点

- **定义/机制**：GPU 拥有大量计算核心，擅长 SIMD/张量并行，特别适合矩阵乘法、向量操作等计算密集任务，而大模型推理的核心算子（如 attention、FFN）天然适合并行化。
- **为什么重要**：相比 CPU，GPU 能大幅加速 LLM 的前向计算；但 Agent Runtime/Agent 工程师不能只关注算力，还要关注显存、带宽和调度。
- **工程做法**：现代 GPU 配合 CUDA/ROCm、Tensor Core、高速 HBM，能在 batch 推理和长上下文场景发挥优势。实际部署时要做算子融合、内存布局优化、batch 调度。
- **权衡/边界**：GPU 算力强但显存和带宽有限；很多推理瓶颈不是“算不动”，而是“装不下”或“搬不动数据”。

#### 一句话版

GPU 适合大模型推理是因为大量核心适合并行化矩阵/向量计算，但受显存和带宽限制。

#### 追问点

- **L3 场景**：如果一个 70B 模型在单卡 GPU 上推理，显存和算力哪个更可能先成为瓶颈？
- **L4 身份/前沿**：在 Agent 多轮 loop 场景中，GPU 的 batch 调度如何影响用户体验和成本？

#### 常见误区

- 只看算力，忽视显存、带宽和 KV Cache 管理。

### Q89. 推理时为什么显存经常比算力更先成为瓶颈？

#### 回答要点

- **定义/机制**：推理时显存被模型权重、KV Cache、中间激活和运行时缓冲共同占用；其中权重和 KV Cache 是主要部分。
- **为什么重要**：长上下文和高并发会迅速放大 KV Cache 占用；很多场景模型能算，但显存放不下，或者访存带宽成为瓶颈。
- **工程做法**：优化方向包括：量化压缩权重、KV Cache 压缩（MLA/GQA/量化）、PagedAttention、上下文裁剪、降低 batch、分布式推理。vLLM 的 PagedAttention 就是典型优化。
- **权衡/边界**：压缩和裁剪可能损失精度；降 batch 影响吞吐；需要在质量、延迟、成本之间取舍。

#### 一句话版

推理时显存常先成瓶颈，因为模型权重和 KV Cache 占用大，长上下文和高并发会快速放大。

#### 追问点

- **L3 场景**：如果显存成为瓶颈，你会优先砍上下文、降 batch，还是做 KV Cache 压缩？
- **L4 身份/前沿**：目标团队 MLA 把 KV Cache 压缩到很小，这种技术对 Agent Runtime 的 Agent 多轮调用有什么实际价值？

#### 常见误区

- 只看算力，忽视显存、带宽和 KV Cache 管理。

### Q90. batch、吞吐、延迟三者之间是什么关系？

#### 回答要点

- **定义/机制**：batch 是同时处理的请求数量；吞吐是单位时间完成的请求数；延迟是单个请求的响应时间。三者相互牵制。
- **为什么重要**：在线 Agent 服务需要低延迟，离线批处理需要高吞吐；Agent Runtime 通常要在二者之间做权衡。
- **工程做法**：增大 batch 通常提升吞吐，但可能增加单请求排队和尾延迟；在线服务常采用动态 batching、continuous batching（vLLM）来兼顾。Agent 产品还要考虑多轮 loop 的累计延迟。
- **权衡/边界**：没有 universally optimal 的 batch 大小；需要根据 SLA、显存、模型特性和请求分布动态调整。

#### 一句话版

更大的 batch 通常提升吞吐但可能增加延迟；在线重延迟、离线重吞吐，需要动态平衡。

#### 追问点

- **L3 场景**：如果 Agent 服务 P99 延迟突然变差，你会先检查 batch 策略还是模型推理本身？
- **L4 身份/前沿**：vLLM 的 continuous batching 对 Agent Runtime 这种多轮、动态 Tool 调用的场景有什么启发？

#### 常见误区

- 只看吞吐或只看延迟，忽视二者的 trade-off 和请求分布的影响。

### Q91. PD 分离（Prefill-Decode disaggregation）对在线 Agent 服务有什么影响？

#### 回答要点

- **定义/机制**：PD 分离把 Prefill（计算密集、吞吐导向）和 Decode（延迟敏感、访存密集）放到不同节点/集群执行，各自独立扩缩容。
- **为什么重要**：在线 Agent 服务既要低 TTFT（Prefill 影响）又要低每个 token 延迟（Decode 影响）；混合部署时两者互相拖累。
- **工程做法**：长 prompt 走 Prefill 集群，生成走 Decode 集群；中间结果通过 KV Cache 传递。Agent Runtime 可以根据任务阶段把 Agent 的多轮调用调度到不同节点。
- **权衡/边界**：增加系统复杂度和 KV Cache 传输成本；小请求可能不适合拆分。

#### 一句话版

PD 分离让 Prefill 和 Decode 各自优化，降低在线 Agent 的 TTFT 和 token 延迟，但增加调度复杂度。

#### 追问点

- **L3 场景**：如果 Agent 的某一步 prompt 很短但生成长度很长，PD 分离的收益大吗？
- **L4 身份/前沿**：在 `长程 Agent 框架` 的长程任务中，是否可以把“理解阶段”放 Prefill 节点、“生成阶段”放 Decode 节点？

#### 常见误区

- 认为 PD 分离对所有请求都有收益，忽视小请求和传输开销。
## 15. AI Coding 与开发者体验

### Q92. 你如何保证 AI 生成代码的质量？

#### 回答要点

- **定义/机制**：AI 生成代码的质量保证不是“生成后看一眼”，而是贯穿需求澄清、测试定义、分步生成、人工 review、回归验证、规则沉淀的完整流程。
- **为什么重要**：AI 擅长生成看似合理的代码，但容易在边界条件、性能、安全和项目隐式约束上出错；没有工程化把关，会直接引入生产风险。
- **工程做法**：先澄清需求和边界，再定义测试/验收标准；让 AI 分步生成，避免一次产出过大；人工重点 review 关键路径、并发、资源管理和错误处理；跑单元测试、集成测试和回归测试；把常见错误沉淀成 checklist 和规则。
- **权衡/边界**：完全人工效率低，完全 AI 风险高；关键是把 AI 放在“草稿和初稿”位置，把人和测试放在“ gatekeeper ”位置。

#### 一句话版

保证 AI 生成代码质量要靠：需求清晰、分步生成、测试先行、人工 review 关键逻辑、回归验证、规则沉淀。

#### 追问点

- **L3 场景**：如果 AI 生成的代码通过了所有测试但在生产环境出错，你会怎么复盘和改进流程？
- **L4 身份/前沿**：AI Coding 的质量保证流程，对 Agent Runtime 的 Eval 和 Trace 设计有什么启发？

#### 常见误区

- 过度依赖 AI，忽视关键逻辑 review、边界条件分析和回归测试。

### Q93. 什么时候 AI Coding 不靠谱？

#### 回答要点

- **定义/机制**：AI Coding 不靠谱的场景通常是“问题空间不确定、约束复杂、错误代价高、需要深度领域知识”的时候。
- **为什么重要**：错误判断 AI 的能力边界会导致技术债、安全漏洞或生产事故。
- **工程做法**：不靠谱的典型场景包括：需求本身不清晰、需要理解大量项目隐含约束、边界条件复杂、错误代价高（金融/安全/一致性）、涉及生产级性能优化、架构级决策、法律合规。这些场景需要人主导，AI 辅助。
- **权衡/边界**：AI 适合“有明确输入输出、模式清晰、可快速验证”的任务；不适合“需要长期上下文、跨系统推理、价值判断”的任务。

#### 一句话版

AI Coding 在需求不清、约束复杂、边界条件多、错误代价高的场景不靠谱，需要人主导。

#### 追问点

- **L3 场景**：如果 AI 生成了一段涉及用户数据处理的代码，你会额外关注哪些安全和合规点？
- **L4 身份/前沿**：在 Agent Runtime 系统开发中，哪些模块可以放心用 AI 辅助，哪些必须人工把控？

#### 常见误区

- 认为 AI 能处理所有编码任务，忽视架构、安全、性能和合规的复杂性。

### Q94. 你怎么把 AI 融入日常研发流程？

#### 回答要点

- **定义/机制**：把 AI 当作“高效率的初级工程师”，在合适的阶段辅助人做拆解、草稿、验证和排障，而不是替代人的判断。
- **为什么重要**：AI 能显著提升研发效率，但只有在清晰的流程和质量 gate 下才能安全落地。
- **工程做法**：方案评审阶段用 AI 做对比和拆解；开发阶段用 AI 写草稿、补测试、生成文档；联调和排障阶段用 AI 分析日志和定位问题；最终由人把控架构、关键逻辑和质量。`长程 Agent 框架` 和数据流水线 Agent 的开发中，可以用 AI 辅助生成 Tool schema、测试用例和文档。
- **权衡/边界**：AI 介入过早可能限制设计思路；介入过晚则效率收益低。需要按任务类型和成熟度分层使用。

#### 一句话版

AI 适合在方案评审、开发草稿、补测试、排障分析中辅助人，最终由人把控架构和质量。

#### 追问点

- **L3 场景**：如果团队有人过度依赖 AI 导致代码质量下降，你会怎么制定使用规范？
- **L4 身份/前沿**：你的 AI Coding 实践，如何反哺 Agent/Agent Runtime 的开发者体验设计？

#### 常见误区

- 把 AI 当成黑盒，不验证输出，不沉淀规则。

### Q95. 什么样的代码库更适合 AI 协作？

#### 回答要点

- **定义/机制**：适合 AI 协作的代码库具有清晰的结构、一致的约定、可运行的测试和明确的边界，让 AI 能在有限上下文中做出正确推断。
- **为什么重要**：AI 的上下文和理解能力有限；代码库越混乱，AI 越容易生成不符合项目规范的代码。
- **工程做法**：适合的特征包括：模块化结构清晰、命名一致、文档和注释适度、测试存在且可运行、Lint/格式化工具完善、提交规范和代码审查流程明确。`长程 Agent 框架` 如果要引入 AI 辅助开发，需要先完善模块边界和测试覆盖。
- **权衡/边界**：过度追求“AI 友好”可能增加不必要的抽象；关键是找到人和 AI 协作的平衡点。

#### 一句话版

结构清晰、命名一致、测试可运行、工具规范明确的代码库更适合 AI 协作。

#### 追问点

- **L3 场景**：如果接手一个老旧代码库，想引入 AI 辅助重构，你会先从哪里入手？
- **L4 身份/前沿**：在 Agent Runtime 的开发者平台中，如何设计 Tool/Skill 让 AI 更容易生成正确的调用代码？

#### 常见误区

- 认为 AI 能适应任何代码库，忽视代码库质量对 AI 输出的影响。

## 16. 算法与 coding

### Q96. 面试中如何快速讲清复杂度？

#### 回答要点

- **定义/机制**：复杂度分析是评估算法随输入规模增长时资源消耗（时间/空间）的变化趋势，通常用大 O 表示最坏或均摊情况。
- **为什么重要**：面试中快速给出复杂度能体现你对算法本质的理解，也是后续优化讨论的基础。
- **工程做法**：先确定核心数据结构（数组、哈希表、堆、树等）；再分析主循环和嵌套关系；分别给出最坏时间复杂度和空间复杂度；如果有剪枝、预处理或均摊分析，单独说明。Agent 系统中复杂度分析直接影响上下文检索、Tool 调用去重等效率。
- **权衡/边界**：平均复杂度和最坏复杂度可能不同；实际性能还受常数、缓存、分支预测影响，不能只看大 O。

#### 一句话版

讲清复杂度要先说数据结构，再说循环嵌套，最后给出最坏时间和空间复杂度，有剪枝或预处理单独说明。

#### 追问点

- **L3 场景**：如果输入规模扩大 100 倍，你的实现还能保持同样复杂度吗？瓶颈会在时间还是空间？
- **L4 身份/前沿**：在 `长程 Agent 框架` 的 Memory 检索或 Tool 调用去重中，复杂度分析如何影响设计选择？

#### 常见误区

- 只背结论不做推导；忽视平均/最坏的区别；不考虑常数因子和实际硬件影响。

### Q97. 字符串格式化类题的高频坑点有哪些？

#### 回答要点

- **定义/机制**：字符串格式化题通常涉及按规则重新组织文本，包括空格、标点、换行、缩进等处理。
- **为什么重要**：这类题看似简单，但边界条件多，容易在细节上出错；也是考察代码细致程度的好题目。
- **工程做法**：高频坑点包括：多空格合并规则、标点前后空格、行长/单词边界、跨行断词、首行/尾行特殊处理、输入规模大时避免反复拷贝大字符串（用 `strings.Builder` 或类似结构）。
- **权衡/边界**：规则越复杂，状态机或分阶段处理越清晰；追求简洁可能漏掉边界。

#### 一句话版

字符串格式化的坑点在空格、标点、行长、断词和大量拷贝，建议用状态机或分阶段处理。

#### 追问点

- **L3 场景**：如果输入规模扩大 100 倍，你会怎么避免反复拼接大字符串？
- **L4 身份/前沿**：在数据流水线 Agent 中，如果要做字幕或文案格式化，这类问题会如何出现？

#### 常见误区

- 只看正常输入，忽视首尾空格、多个连续空格、空输入等边界。

### Q98. digit set / digit mask 统计题的核心思路是什么？

#### 回答要点

- **定义/机制**：digit set / digit mask 是用位掩码表示一个数字中出现了哪些十进制位（如 112 出现数字 1 和 2，掩码为 `0b110`）。
- **为什么重要**：这类技巧能把“数字集合相等”的问题转化为“位掩码相等”，从而用哈希表做 O(n) 统计。
- **工程做法**：对每个数计算其 digit mask；用哈希表统计每个 mask 出现次数；用组合数 `C(n,2)` 计算相同 mask 的数对数量；最后转成概率或频率。注意前导零和重复数字的处理。
- **权衡/边界**：mask 空间小（最多 1024 种）时非常高效；但如果需要区分数字出现次数，mask 就不够，需要额外信息。

#### 一句话版

用位掩码表示数字中出现了哪些位，统计相同掩码的出现次数，再用组合数计算对数。

#### 追问点

- **L3 场景**：如果题目要求“数字集合完全相同且每个数字出现次数也相同”，mask 方法还适用吗？你会怎么改？
- **L4 身份/前沿**：这种位掩码统计思想，在 Agent 的状态压缩或特征去重中有什么应用场景？

#### 常见误区

- 混淆“数字集合相同”和“数字完全相同”；忽视前导零或重复数字的影响。

### Q99. BFS 变种题应该先想什么？

#### 回答要点

- **定义/机制**：BFS 变种通常用于在状态空间中寻找最短路径或最少步骤，关键是把问题抽象成图：状态是节点，合法操作是边。
- **为什么重要**：BFS 变种覆盖了迷宫、密码锁、游戏状态、字符串变换等一大类问题；想清楚状态定义就成功了一半。
- **工程做法**：先想状态是什么；一步操作如何定义；状态如何扩展（边）；是否需要去重/访问集合；是否满足最短路性质；状态空间大时考虑双向 BFS 或 A*。
- **权衡/边界**：BFS 保证最短但状态空间可能爆炸；需要根据问题特点选择普通 BFS、双向 BFS、0-1 BFS 或带权最短路径。

#### 一句话版

BFS 变种先想：状态是什么、一步操作是什么、怎么扩展、怎么去重、是否满足最短路。

#### 追问点

- **L3 场景**：如果状态空间很大但目标状态已知，你会考虑双向 BFS 吗？它的适用条件是什么？
- **L4 身份/前沿**：在 Agent 的规划或搜索中，BFS/DFS/A* 的选择依据是什么？

#### 常见误区

- 状态定义不清就写 BFS；忽视去重导致指数级爆炸；该用 DFS/DP 的场景强行 BFS。

### Q100. 写完代码后你会先检查什么？

#### 回答要点

- **定义/机制**：写完代码后的检查是防止低级错误、确保鲁棒性的必要步骤，不只是跑测试。
- **为什么重要**：面试中主动检查边界能体现工程素养；实际开发中能减少线上 bug。
- **工程做法**：检查输入边界（空、单元素、极大/极小值）；空数据和默认值；重复元素或重复访问；整数溢出；时间复杂度是否真满足约束；资源释放；并发安全（如果相关）。
- **权衡/边界**：检查过多会拖慢开发；关键是形成固定 checklist，覆盖最常见的失败模式。

#### 一句话版

写完代码先检查边界、空值、极值、重复、溢出、复杂度，再跑测试。

#### 追问点

- **L3 场景**：如果面试官说“你的代码在极端输入下会超时”，你会从哪几个角度排查？
- **L4 身份/前沿**：在 `长程 Agent 框架` 的 Tool 调用链路中，类似的“写完先检查”习惯如何应用到参数校验和错误处理？

#### 常见误区

- 只跑一两个正常用例就以为对了，不系统检查边界和异常。

### Q101. 双指针 / 滑动窗口类题的高频写法是什么？

#### 回答要点

- **定义/机制**：双指针用两个索引同向或反向遍历数组/链表；滑动窗口是双指针的特例，通过维护 `[left, right)` 区间来满足某种条件（和、字符种类数、最大值/最小值约束）。
- **为什么重要**：这类技巧能把 O(n²) 的枚举降到 O(n)，是字符串处理、子数组/子串问题、日志清洗、流式数据中最常用的优化手段之一。
- **工程做法**：高频模式包括：1）同向快慢指针（去重、找环、partition）；2）反向双指针（两数之和、合并有序数组）；3）可变窗口（右扩到满足条件，左缩到刚好不满足，记录最优）。写代码时注意 `while` 循环的终止条件和窗口收缩时的状态清理。
- **权衡/边界**：双指针要求数据有序或问题具有单调性；滑动窗口维护的状态必须能快速更新，否则缩窗成本会退化成 O(n²)。

#### 一句话版

双指针/滑动窗口通过维护一个或两个索引把暴力枚举降为线性，关键是找到问题的单调性并保证窗口状态可快速更新。

#### 追问点

- **L3 场景**：如果滑动窗口内需要维护最大值，你会用单调队列还是每次遍历？ trade-off 是什么？
- **L4 身份/前沿**：在 `长程 Agent 框架` 的流式事件处理或数据流水线 Agent 的弹幕清洗中，滑动窗口思想可以怎么用？

#### 常见误区

- 窗口收缩时忘记更新统计量；把 `left++` 和状态更新顺序写反；没有处理空窗口或全不满足条件的边界。

### Q102. 前缀和 / 差分适合解决什么问题？

#### 回答要点

- **定义/机制**：前缀和 `prefix[i] = sum(a[0..i-1])`，用于 O(1) 查询区间和；差分 `diff[i] = a[i] - a[i-1]`，用于 O(1) 区间加减，最后做一次前缀和还原原数组。
- **为什么重要**：区间查询和区间更新是 coding 面试和工程系统中的高频问题（如流量统计、实验指标累计、配置生效区间），前缀和/差分能把多次查询或更新的复杂度降下来。
- **工程做法**：一维前缀和解决子数组和查询；二维前缀和解决子矩阵和；差分数组解决“对区间 [l,r] 同时加 k”的批量更新。注意下标从 0 还是从 1 开始，以及边界是否包含。
- **权衡/边界**：前缀和适合静态数组或更新较少的场景；频繁单点更新时需要树状数组或线段树。差分不适合需要随时查询中间结果的场景。

#### 一句话版

前缀和把区间查询变成 O(1)，差分把区间更新变成 O(1)；两者都依赖下标约定和边界处理。

#### 追问点

- **L3 场景**：如果数组会频繁单点更新且需要频繁区间查询，前缀和不够用了，你会换什么数据结构？
- **L4 身份/前沿**：在 Agent Runtime 的 Eval 指标统计或 Agent 调用成本分桶中，前缀和/差分思想有什么应用场景？

#### 常见误区

- 下标从 0/1 混淆导致差一错误；差分后忘记最后做前缀和还原；把适合线段树的动态更新场景硬用前缀和。

### Q103. 链表题最容易犯的错误是什么，如何避免？

#### 回答要点

- **定义/机制**：链表通过指针串联节点，插入/删除只需要改指针，不需要移动数据；但指针改错会导致断链、环、内存泄漏或访问已释放节点。
- **为什么重要**：链表题考察对指针和边界的细致程度，也是理解更复杂结构（LRU、跳表、链式前向星）的基础。
- **工程做法**：最易犯的错误和处理方式：1）空指针访问——先用 dummy 头节点统一边界；2）断链——先保存 next 再改指针；3）头节点特殊处理——用 dummy 头避免分支；4）内存泄漏/重复释放——明确每个指针的生命周期；5）成环——检查尾节点是否指向 nullptr。
- **权衡/边界**：dummy 头节点增加一个节点空间，但能显著减少分支判断；递归解法简洁但栈深度受限制。

#### 一句话版

链表题最容易断链、空指针访问和头节点特殊处理；用 dummy 头和先保存 next 再改指针能避免大部分错误。

#### 追问点

- **L3 场景**：如果要求 O(1) 空间反转链表，你会用迭代还是递归？各自的边界条件是什么？
- **L4 身份/前沿**：在 `长程 Agent 框架` 的异步任务链或事件队列中，链表/队列的边界处理有哪些类似经验？

#### 常见误区

- 没保存 `next` 就改 `cur->next` 导致断链；忘记处理空链表或单节点链表；递归反转链表时没正确拼接新头。

### Q104. 递归改迭代时最常踩的坑是什么？

#### 回答要点

- **定义/机制**：递归利用系统调用栈保存状态；迭代改递归需要显式用栈（或队列）保存待处理节点和中间状态，模拟调用栈的行为。
- **为什么重要**：工程系统中递归深度可能溢出、不可控（如 C++ 默认栈很小），或者需要更好的性能和内存可控性；掌握递归改迭代是避免线上栈溢出和写出非阻塞代码的关键。
- **工程做法**：常见模式：1）树/图的 DFS/BFS——用显式 stack/queue；2）尾递归——直接改成循环；3）需要回溯的分支递归——栈里保存 (node, state, index) 三元组；4）分治——用栈保存子问题边界。关键是区分“哪些状态必须保留、哪些可以复用”。
- **权衡/边界**：迭代写法通常更省栈空间、更可预测，但代码更长、状态管理更复杂；递归更适合思路表达，但深度大时有栈溢出风险。

#### 一句话版

递归改迭代的核心是用显式栈保存“调用栈状态”，最容易踩的坑是漏保存回溯状态或错误恢复现场。

#### 追问点

- **L3 场景**：如果一棵二叉树深度可能达到 10^5，你会怎么选择遍历方式？为什么？
- **L4 身份/前沿**：在 Agent 的搜索/规划算法中，递归深度爆炸和状态空间爆炸分别是哪两类问题，如何分别应对？

#### 常见误区

- 只压节点不压状态，导致无法回溯；迭代结束条件写错；把递归的“后序”逻辑硬套成前序栈写法。

## 18. Transformer 架构深度

### Q107. MHA、MQA、GQA、MLA 的演进逻辑是什么？

#### 回答要点

- MHA：每个 head 独立 Q/K/V，KV Cache 最大。
- MQA：所有 head 共享一组 K/V，KV Cache 最小但质量下降。
- GQA：G 个 Q head 共享一组 K/V，平衡质量和效率（Llama 2 70B 使用）。
- MLA：低秩压缩 K/V 到 latent space，KV Cache 压缩约 93%。
- 演进动力始终是 KV Cache 的显存和带宽压力。

### Q108. RoPE 旋转位置编码的原理是什么？

#### 回答要点

- 对 Q 和 K 的相邻维度对施加旋转矩阵，旋转角度与位置成正比。
- 核心优势：相对位置编码——两个 token 的 attention score 只依赖相对距离。
- 外推性好：训练时见过的长度之外的序列也能处理（虽然效果会衰减）。
- 大多数现代 LLM 使用 RoPE（Llama、Qwen 等）。

### Q109. MoE（Mixture of Experts）在 Agent 场景中有什么特殊考虑？

#### 回答要点

- Agent 通常需要大模型做复杂推理，MoE 可以用更少的激活参数达到更大模型的效果。
- MoE 的推理延迟主要受 all-to-all 通信和 expert 权重加载影响。
- Agent 场景中 prefill 很长（system prompt + 工具描述），MoE 的通信开销占比更高。
- 路由策略直接影响不同任务类型的质量一致性。

### Q110. FlashAttention 为什么对 Agent 场景特别重要？

#### 回答要点

- Agent 的 system prompt 通常很长（工具描述、few-shot examples），attention 计算量大。
- FlashAttention 减少 HBM 访问，在长序列下加速效果更显著。
- Agent 的 decode 阶段也很长（多步推理），attention kernel 的调用频率高。
- 是推理引擎中 attention 后端的标配。

### Q111. PagedAttention 的 Copy-on-Write 机制是什么？

#### 回答要点

- 当多个请求共享相同前缀时，不复制 KV Cache block，而是增加引用计数。
- 只在某个请求要修改 block 时才真正复制（类似操作系统的 COW）。
- 在 Agent 场景中 system prompt 固定且长，COW 可以显著减少显存占用。
- 配合 Prefix Caching 使用效果更好。

### Q112. 位置编码外推（YaRN、PI）是什么？为什么 Agent 场景需要？

#### 回答要点

- 模型训练时的上下文长度有限，但 Agent 可能需要处理更长的对话历史。
- YaRN：对 RoPE 的频率做插值，让模型能处理更长序列。
- PI（Position Interpolation）：把位置索引压缩到训练范围内。
- Agent 场景中多轮对话累积的上下文可能超出训练长度，需要外推支持。

### Q113. Attention 的 O(N²) 复杂度对 Agent 有什么影响？

#### 回答要点

- Agent 的 system prompt 通常是几千到几万 token，加上多轮对话，N 很大。
- O(N²) 意味着 prefill 延迟随上下文长度平方增长，TTFT 可能达到数秒。
- 解决方向：FlashAttention（IO 优化）、滑动窗口、稀疏注意力、线性注意力。
- 在 Agent Infra 设计中，上下文管理直接影响用户体验。

### Q114. LayerNorm 和 RMSNorm 的区别？为什么 Agent Runtime 需要关注？

#### 回答要点

- LayerNorm：均值中心化 + 缩放。
- RMSNorm：只缩放（去掉均值中心化），计算更快。
- 大多数现代 LLM 使用 RMSNorm（Llama、Qwen 等）。
- Agent Runtime 做推理时需要确保 Norm 实现和训练一致，否则精度受损。

### Q115. 残差连接在深层 Transformer 中的作用？

#### 回答要点

- 缓解梯度消失，让信息能跳过中间层直接传递。
- Pre-Norm vs Post-Norm：Pre-Norm 训练更稳定，现代 LLM 普遍使用。
- 深层模型（100+ 层）可能需要额外的归一化策略。
- Agent 使用的大模型通常有 32-128 层，残差连接是训练稳定性的基础。

### Q116. SwiGLU 和 GELU 的区别？为什么现代 LLM 选择 SwiGLU？

#### 回答要点

- GELU：标准激活函数，近似 ReLU 的平滑版本。
- SwiGLU：引入门控机制（gate × value），表达力更强。
- SwiGLU 的 FFN 有 3 个 Linear 层（gate、up、down），参数量多 50%。
- 效果更好，现代 LLM（Llama、Qwen 等）普遍使用。

## 19. Agent 评估与 Benchmark

### Q117. SWE-bench 的评估流程是怎样的？为什么难？

#### 回答要点

- 流程：给 Agent 一个真实 GitHub issue → Agent 在代码仓库中定位问题、修改代码 → 运行测试用例验证。
- 难点：真实代码库规模大、依赖复杂、需要理解 issue 和写正确补丁。
- 对 Agent 的长上下文、代码理解、工具使用要求极高。
- SWE-bench Verified 是人工验证可解的子集，噪声更低。

### Q118. GAIA 和 WebArena 主要测什么能力？

#### 回答要点

- GAIA：测多步推理和外部知识整合，需要搜索、计算、信息综合。
- WebArena：测真实网站操作，需要浏览器环境、动态网页交互、状态跟踪。
- GAIA 偏认知推理，WebArena 偏操作执行。
- 两者都需要完整的沙箱环境和结果判定机制。

### Q119. Agent 评估框架和传统 LLM 评估有什么区别？

#### 回答要点

- 传统 LLM 评估：单轮 prompt → 文本输出 → 文本匹配/打分。
- Agent 评估框架：多轮交互 + 环境状态 → 动作序列 → 环境反馈/任务完成度。
- 需要沙箱环境（Docker/浏览器/API mock）保证可复现。
- 评估维度更广：规划、工具使用、反思纠错、成本效率。

### Q120. LLM-as-Judge 有什么缺点？怎么减少偏见？

#### 回答要点

- 缺点：位置偏见、长度偏见、格式偏见、标准不一致。
- 减少偏见：交换位置多次投票、多 judge 模型投票、结构化评分标准（rubric）。
- 用人类标注校准 judge。
- 让 judge 先评价再比较，而不是直接比较。

### Q121. 如何保证 Agent 评估的可复现性？

#### 回答要点

- 固定随机种子、锁定模型版本。
- 容器化沙箱（Docker）固定 OS、依赖、工具版本。
- 固定数据集和配置版本。
- 记录完整 trajectory 和环境状态。
- 使用相同的 judge prompt。

## 20. Agentic 推理优化

### Q122. Agent 场景的 Prefill 和 Decode 有什么特殊性？

#### 回答要点

- Prefill 很长：system prompt（工具描述、few-shot）通常数千到数万 token。
- Decode 也很长：Agent 需要多步推理，每步可能生成几百 token。
- 前缀复用率高：同一 Agent 会话中 system prompt 不变。
- 对 TTFT 和 TBT 都敏感：用户等待时间直接影响体验。

### Q123. Prefix Caching 在 Agent 场景中怎么复用 system prompt？

#### 回答要点

- Agent 的 system prompt 通常固定且长（工具描述、规则、few-shot）。
- Prefix Caching 缓存 system prompt 的 KV Cache，后续请求直接复用。
- vLLM 的 APC 是 block-level，SGLang 的 RadixAttention 是 token-level（更精细）。
- 在前缀复用率高的场景下，TTFT 可以从 2.5s 降到 1.2s。

### Q124. DualPath 在 Agentic 推理中解决什么问题？

#### 回答要点

- PD 分离架构下，KV Cache 需要从 Prefill 节点传输到 Decode 节点。
- Agent 的 KV Cache 很大（长 system prompt + 多轮对话），传输带宽是瓶颈。
- DualPath 用两条路径并行加载 KV Cache，聚合带宽。
- 关键优化：分块传输、预取、与计算重叠。

## 21. 每日抽查清单

每天随机抽 10 题，按 L1-L4 分层自查：

- **L1（30 秒版）**：能快速给出结论，不卡壳。
- **L2（2 分钟版）**：能给出结构化回答：定义/机制 → 为什么重要 → 工程做法 → 权衡/边界。
- **L3（场景版）**：能举出一个自己的项目例子（`长程 Agent 框架`、数据流水线 Agent、策略平台 / 规则引擎 / MCP Tool），并回答一条场景追问。
- **L4（身份/前沿版）**：能把问题焊接到 目标团队 技术栈（MLA、GRPO、DualPipe、Mooncake、vLLM），提出自己的判断。

每题还要检查：

- 能说出一个 trade-off
- 能补一个失败模式或常见坑
- 不会被一个尖锐追问问住
