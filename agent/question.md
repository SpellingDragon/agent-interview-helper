# Agent 工程师高频问答题库

> 适用对象：Agent 平台 / AI Infra 方向候选人
> 目标：把面试中的高频问题准备成可直接口述的题库
> 使用方式：
> 1. 先自己口头回答题目
> 2. 再看“回答要点”检查是否漏项
> 3. 最后补成 1 分钟版和 3 分钟版

---

## 1. 使用说明

建议每道题至少准备三层表达：

- `30 秒版`：先给结论，适合快问快答
- `2 分钟版`：给定义、场景、权衡
- `追问版`：补充失败模式、工程细节、 trade-off

回答时优先使用以下结构：

1. 先定义问题
2. 再说明为什么重要
3. 再讲工程上的常见做法
4. 最后讲边界和权衡

## 2. 岗位匹配与动机

### Q1. 你为什么适合 Agent 平台团队？

#### 回答要点

- 过去长期做复杂策略系统、规则引擎、DAG Runtime、DSL、SDK、实验配置，这些经验和 Agent Runtime 的核心抽象高度相关。
- 不只是做平台工程，也有真实 Agent 实践，包括 长程 Agent 框架、数据流水线 Agent、MCP Tool、Memory 驱动上下文管理、Trace。
- 是 AI Coding 和 Agent 产品的深度用户，能从工程建设者和开发者体验两个视角看问题。
- 对模型、工具、上下文、反馈闭环之间的关系有工程化理解，而不只是会接模型 API。

#### 一句话版

我适合这个岗位，不是因为我刚开始接触 Agent，而是因为我本来就在做复杂运行时和策略系统，现在只是把这套能力迁移到 Agent Runtime。

### Q2. 你为什么想做 Agent Runtime，而不是纯模型或纯应用？

#### 回答要点

- 纯模型更关注能力边界，纯应用更关注业务闭环，而 Agent Runtime 正好处于两者之间。
- 自己最擅长的是把不稳定、快速变化的能力组织成可配置、可观测、可灰度、可回滚的系统。
- Agent Runtime 的核心问题，如上下文组织、工具调用、记忆管理、执行观测、反馈闭环，和过去做复杂系统的经验高度一致。

### Q3. 你理解的 Agent Runtime 是什么？

#### 回答要点

- Agent Runtime 不只是一个 Agent SDK，而是把模型能力稳定转化为真实任务执行能力的运行时和工程系统。
- 它至少要解决五类问题：上下文管理、工具调用、状态与记忆、执行可观测、反馈闭环。
- 没有 Runtime，模型只能“回答”；有了 Runtime，模型才能“完成任务”。

### Q4. 你为什么认为这个岗位需要广度而不是单点技能？

#### 回答要点

- Agent Runtime 横跨模型、运行时、工具系统、后端基础、开发者体验和产品迭代。
- 真实问题往往不是单点 bug，而是上下文、工具、模型和策略耦合后的综合问题。
- 团队又要求和研究员、模型训练工程师、产品经理协作，必须能跨语言沟通。

## 3. 项目与经历

### Q5. 你做的长程 Agent 框架，核心探索点是什么？

#### 回答要点

- 基于 Go 做轻量级长程 Agent 框架，核心不是再造一个聊天壳，而是探索长任务中的 Memory、上下文注入、多 Agent 协作和执行可观测。
- 重点抽象是：Agent、Model、Tool、Skill、Memory、Trace。

### Q6. 长程 Agent 框架 中 Skill 和 Tool 的区别是什么？

#### 回答要点

- Tool 更偏原子能力，例如查询、上传、解析、执行。
- Skill 更像可复用的任务模板或策略块，通常会组合多步行为、约束调用方式或定义角色边界。
- Tool 解决“能做什么”，Skill 解决“这类任务通常怎么做”。

### Q7. 数据流水线 Agent 中，哪些步骤是 Agent 决策，哪些是工程固定？

#### 回答要点

- 事件监听、直播流获取、ffmpeg 处理、投稿接口调用等偏工程和确定性步骤，固定在 Tool 或工作流中。
- 哪些互动片段值得分析、如何组合已有工具、是否需要做多模态理解、更适合哪种切片方案，这部分由 Agent 规划。
- 原则是把确定性交给工程，把不确定性留给模型。

### Q8. 数据流水线 Agent 最难的工程问题是什么？

#### 回答要点

- 长链路自动化中间状态管理复杂，包含事件采集、抽样、分析、多模态理解、剪辑和投稿。
- 上下文对象很大，不能直接塞给模型，需要把大对象和模型上下文解耦，通过 Tool 侧按需取数。
- 多模态链路耗时长、失败点多，需要可追踪和可重试。

### Q9. 策略平台 / 规则引擎和 Agent Runtime 有什么关系？

#### 回答要点

- 规则引擎的对象是“策略”，Agent Runtime 的对象是“模型驱动行为”，但两者都要解决抽象表达、运行时执行、灰度回滚、校验、可观测、实验闭环。
- 项目里把模型、prompt、tool 的策略从代码迭代抽成配置热变更，这本质上已经在做 Agent Runtime 的一部分。
- 可以理解为从传统业务策略系统，向模型行为策略系统的自然延伸。

### Q10. 为什么把模型策略从代码发布改成配置热变更？

#### 回答要点

- 降低实验迭代成本，提高策略调整速度。
- 让模型、prompt、tool 策略和实验平台、审批、回滚、灰度体系更好融合。
- 避免业务逻辑和实验逻辑深度耦合。

### Q11. 为什么把内容解析能力以 MCP 形式提供？

#### 回答要点

- MCP 能把多媒体资源解析能力标准化暴露出来，降低不同宿主和上层 Agent 的接入成本。
- 协议统一后，更容易做缓存、权限、复用和观测。
- 这类能力天然适合作为通用工具，而不应该嵌死在单一业务服务里。

## 4. Agent Runtime 核心

### Q12. ReAct 是什么？为什么工程上常用它？

#### 回答要点

- ReAct 是模型在推理和行动之间交替的一种模式，通常体现为“思考 -> 调用工具 -> 观察结果 -> 继续思考”。
- 它适合需要外部信息、工具执行和多步决策的任务。
- 工程上好处是每一步都可观测、可插桩、可约束。

### Q13. Agent Loop 的核心状态有哪些？

#### 回答要点

- 接收任务
- 生成下一步决策
- 选择或调用工具
- 获取观察结果
- 更新状态和上下文
- 判断完成或进入异常处理

### Q14. 为什么说 Agent 不等于多轮聊天？

#### 回答要点

- 多轮聊天只是连续对话，未必涉及外部动作和状态管理。
- Agent 需要任务驱动、工具执行、状态演进、失败恢复和目标达成。
- 关键不在轮数，而在是否具备闭环执行能力。

### Q15. Tool Use 的完整流程是什么？

#### 回答要点

- 声明可用工具及参数 schema
- 模型根据目标选择工具
- 生成结构化参数
- 工程侧做反序列化和校验
- 执行工具并获取结果
- 将结果回填给模型继续决策
- 最终形成输出或进入下一轮

### Q16. Tool schema 为什么重要？

#### 回答要点

- 它是模型和工程系统之间的契约。
- schema 越清晰，模型越容易稳定地产出正确参数。
- 还能支撑校验、错误提示、版本兼容和调试。

### Q17. Tool 设计最常见的失败模式有哪些？

#### 回答要点

- 模型选错工具
- 参数缺失或字段幻觉
- 非幂等工具被错误重试
- 返回结果过大，污染上下文
- 工具内部依赖下游超时或失败
- 工具结果本身正确，但不符合用户真正目标

### Q18. Tool 返回太长怎么办？

#### 回答要点

- 不要把全量原始结果直接回填模型。
- 优先做结构化摘要、分层返回、分页、ID 引用或 key 引用。
- 必要时只返回可供下一步决策的最小信息集。

### Q19. Skill 和 Workflow 有什么区别？

#### 回答要点

- Workflow 更强调流程和依赖，通常由工程预定义。
- Skill 更偏任务模式的封装，既可以是半结构化策略，也可以携带调用偏好和约束。
- Workflow 更确定，Skill 更贴近“如何做这类事情”的经验沉淀。

### Q20. 什么情况下应该做 Multi-Agent？

#### 回答要点

- 任务天然可拆分，且子任务之间边界清晰。
- 不同子任务需要不同模型、工具或能力边界。
- 上下文隔离收益足够大，能显著降低复杂度或成本。
- 子任务可以独立评估和调试。

### Q21. 为什么很多 Multi-Agent 系统效果并不好？

#### 回答要点

- 只是机械分角色，没有真正的信息隔离和专业化收益。
- 多了通信和协作成本，token 和延迟显著上升。
- 中间状态和协议设计复杂，调试困难。
- 错误会在子 Agent 间传递和放大。

### Q22. 单 Agent 和 Multi-Agent 你怎么选？

#### 回答要点

- 默认先从单 Agent 做起，除非任务分解天然成立。
- 当上下文过大、角色冲突明显、工具边界不同、需要独立评估子任务时，再考虑多 Agent。
- 是否引入 Multi-Agent，取决于收益是否大于复杂度。

## 5. MCP / Tool 平台

### Q23. MCP 和普通函数调用相比，工程价值是什么？

#### 回答要点

- MCP 把工具和资源访问标准化，降低集成成本。
- 更容易实现跨宿主、跨语言和跨团队复用。
- 有助于统一工具发现、权限、观测和版本管理。

### Q24. 什么时候一个能力更适合做 MCP Tool？

#### 回答要点

- 能力通用、边界清晰、输入输出可结构化。
- 希望被多个 Agent、多个宿主或多个场景复用。
- 需要独立治理权限、缓存、可靠性和观测。

### Q25. MCP 接入后还需要做哪些工程工作？

#### 回答要点

- 参数校验和错误映射
- 权限和资源隔离
- 调用超时、重试、熔断
- 结果裁剪和缓存
- trace、日志和指标

### Q26. 你如何评价 Tool 平台的好坏？

#### 回答要点

- 可发现性是否强
- schema 是否清晰稳定
- 调试是否方便
- 是否有足够可观测性
- 是否易于接入、治理和版本演进

## 6. Memory / Context Engineering

### Q27. Prompt、Context、Memory 的区别是什么？

#### 回答要点

- Prompt 更偏当前交互中的指令表达。
- Context 是当前这次推理实际喂给模型的全部信息。
- Memory 是跨轮次、跨阶段存储和再利用的信息抽象。
- Memory 不一定直接进入 prompt，通常需要检索、筛选和压缩后才能进入 context。

### Q28. Memory 和 RAG 的区别是什么？

#### 回答要点

- RAG 更偏从外部知识源检索相关内容补充当前回答。
- Memory 更偏系统对历史偏好、状态、任务进度、工具结果和世界状态的持续维护。
- 二者可以重叠，但 Memory 的范围更广，强调持续状态而不只是知识检索。

### Q29. 为什么不能把聊天历史全都当 Memory？

#### 回答要点

- 历史记录里噪声很多，并不都适合长期保留。
- 直接堆历史会导致上下文膨胀和污染。
- Memory 的价值在于被抽象、清洗、分层和按需取用，而不是无差别堆积。

### Q30. 长期记忆系统最常见的问题是什么？

#### 回答要点

- 写入过多，检索命中率低
- 过期信息未清理
- 错误信息被长期固化
- 语义相近但不相关的信息干扰当前任务
- 没有分层和生命周期管理

### Q31. 你如何设计 Memory 的写入策略？

#### 回答要点

- 不是每轮都写，优先写稳定、高价值、跨阶段有用的信息。
- 按类型分层，例如用户偏好、任务状态、工具结果、世界状态。
- 明确生命周期，区分临时记忆和长期记忆。
- 写入前做清洗、去重和摘要。

### Q32. 你如何设计 Memory 的读取策略？

#### 回答要点

- 先判断当前任务阶段需要什么信息。
- 按阶段和类型筛选，不做全量注入。
- 优先注入当前决策真正依赖的最小集合。
- 必要时使用 key 引用而不是原文注入。

### Q33. 你如何判断哪些上下文应该进 prompt？

#### 回答要点

- 看当前任务目标是否需要它
- 看它是否能改变当前决策
- 看它是否比 token 成本更有价值
- 看是否会引入歧义、冲突或噪声

### Q34. 为什么说 Context Engineering 是 Agent Runtime 的核心？

#### 回答要点

- 模型性能上限不仅取决于模型本身，还取决于你给了什么上下文。
- 很多失败不是模型不会，而是喂错了信息。
- Context Engineering 决定了信息组织、阶段注入、长度控制和冲突消解，是稳定性和成本的关键。

## 7. Trace / Eval / Replay

### Q35. Trace 和日志有什么区别？

#### 回答要点

- 日志更偏离散事件记录。
- Trace 更强调一次任务执行链路上的跨阶段关联关系。
- 在 Agent 场景中，Trace 需要把模型输入输出、工具调用、状态变化和关键上下文串起来。

### Q36. Agent Trace 至少应该记录什么？

#### 回答要点

- 用户输入
- 系统 prompt
- 当前上下文切片
- 模型请求和响应
- 工具调用参数与结果
- 重试、异常和终止原因
- 最终输出

### Q37. 为什么 Replay 很重要？

#### 回答要点

- 方便复现问题
- 便于比较 Prompt、Policy、Model 或 Tool 变更的影响
- 可以做回归测试和离线评估
- 是调试复杂 Agent 系统的关键能力

### Q38. Eval 应该评什么？

#### 回答要点

- 任务成功率
- 工具成功率
- 平均步数
- 重试率
- 时间成本
- token 成本
- 用户满意度 proxy
- 人工标注质量

### Q39. 没有标准答案的 Agent 任务怎么做 Eval？

#### 回答要点

- 用多维指标替代单一正确率。
- 结合结构化检查、规则检查、人工抽检和用户反馈。
- 评估过程本身也要分层，先评任务完成，再评路径质量和成本。

### Q40. 如何定位是模型问题、工具问题还是上下文问题？

#### 回答要点

- 看模型是否在无工具条件下也无法给出合理决策。
- 看工具调用参数和返回结果是否正确。
- 看上下文是否遗漏关键信息，或被噪声干扰。
- 配合 trace 和 replay 做定位，不能只看最终失败。

## 8. LLM 与推理基础

### Q41. Transformer 的基本组成是什么？

#### 回答要点

- token embedding
- self-attention
- feed-forward network
- 残差连接和层归一化
- 多层堆叠形成深层表示

### Q42. Attention 的直觉是什么？

#### 回答要点

- 当前 token 在生成表示时，不是只看自己，而是对历史 token 做加权聚合。
- 不同 token 对当前决策的重要程度不同，Attention 就是在学习这种相关性。

### Q43. KV Cache 的作用是什么？

#### 回答要点

- 在 decode 阶段复用历史 token 的 key 和 value，避免每生成一个新 token 都重复计算全部历史。
- 它显著降低重复计算，提高推理效率。
- 代价是会占用显存，而且上下文越长，占用越大。

### Q44. Prefill 和 Decode 有什么区别？

#### 回答要点

- Prefill 是先把已有上下文整段编码进去，通常计算密集、偏吞吐。
- Decode 是逐 token 生成，强依赖 KV Cache，延迟敏感、并发受限。
- 长任务里 decode 常常更难优化。

### Q45. 长上下文为什么昂贵？

#### 回答要点

- prefill 阶段要处理更多 token
- KV Cache 会更大，占更多显存
- 后续 decode 每一步都要访问更长历史
- 容易引入无关信息，影响质量

### Q46. 显存主要消耗在哪？

#### 回答要点

- 模型权重
- KV Cache
- 中间激活和运行时缓冲
- batch 和并发带来的额外占用

### Q47. 显存不够时你会考虑哪些方向？

#### 回答要点

- 缩短上下文或做更严格的 context engineering
- 降低 batch 或并发
- 换更小模型或分层路由
- 使用量化、paged KV 或分布式推理方案
- 对工具结果和历史做摘要或外置存储

### Q48. 为什么 Agent 特别需要关注推理成本？

#### 回答要点

- Agent 不是一次调用，而是多轮 loop，成本会放大。
- 工具结果、历史轨迹和中间状态很容易把上下文做大。
- 复杂任务既要质量，也要延迟、稳定性和成本平衡。

### Q49. 采样参数通常影响什么？

#### 回答要点

- 温度和 top-p 影响输出的随机性和多样性。
- 在需要稳定结构化输出时，通常倾向于更保守的采样。
- 在头脑风暴、候选方案生成时可以适当放开。

### Q50. 函数调用 / 工具调用和普通文本生成的差异是什么？

#### 回答要点

- 结构化输出要求更高。
- 参数 schema 和错误恢复比自由文本更关键。
- 需要把模型输出纳入工程执行链路，而不是只看自然语言质量。

## 9. Multi-Agent / Planning

### Q51. Planning 一定要交给模型吗？

#### 回答要点

- 不一定。
- 规划可以完全由模型做，也可以完全由工程工作流做，很多时候更适合做混合式。
- 如果流程约束很强，工程显式编排往往更稳定。

### Q52. 什么时候该让工程约束流程，什么时候让模型自由规划？

#### 回答要点

- 已知流程、可验证、错误代价高的部分，交给工程。
- 目标开放、路径不确定、需要探索的部分，让模型决策。
- 经验上是“确定性归工程，不确定性归模型”。

### Q53. Agent 规划失败时常见原因有哪些？

#### 回答要点

- 目标理解错
- 工具能力边界不清
- 上下文缺失或污染
- 中间结果未结构化
- 没有阶段性校验和纠偏

### Q54. 你如何看待 Subagent？

#### 回答要点

- 它适合封装子任务、隔离上下文和角色边界。
- 如果只是为了“显得高级”而拆分，会增加复杂度。
- 是否引入 Subagent，要看调试性和收益是否更好。

## 10. 规则引擎 / DAG Runtime / DSL

### Q55. 规则引擎最重要的设计点是什么？

#### 回答要点

- 表达能力和可控性的平衡
- 执行效率
- 冲突处理和优先级
- 校验、灰度、回滚
- 可观测和可解释

### Q56. DSL 为什么容易失控？

#### 回答要点

- 表达力一旦过强，就可能接近通用编程语言，带来理解和治理成本。
- DSL 过于灵活时，校验难、测试难、可视化难。
- 所以 DSL 设计通常要在灵活性和可验证性之间做取舍。

### Q57. DAG Runtime 和 Agent Runtime 的异同是什么？

#### 回答要点

- 相同点是都要处理依赖、状态、执行、可观测和失败恢复。
- 不同点是 DAG 更确定，Agent Runtime 的路径更开放、更依赖模型输出。
- 因此 Agent Runtime 更需要 trace、guardrail、上下文管理和反馈闭环。

### Q58. 为什么 SDK 本地预解析配置能提效？

#### 回答要点

- 避免运行时反复做反序列化和语法解析。
- 提升热点路径性能。
- 更容易做启动期校验和结构化缓存。

### Q59. 配置热更新最大的风险是什么？

#### 回答要点

- 错误配置直接进入线上
- 新旧版本行为不一致
- 配置和代码契约失配
- 缺少灰度、审批、回滚和校验会把风险放大

## 11. 后端与系统设计

### Q60. 如何保护一个被 Agent 高频调用的 Tool 服务？

#### 回答要点

- 做超时、重试、熔断和限流
- 做场景级和内容级缓存
- 区分只读和有副作用工具
- 做幂等保护和降级路径
- 用 trace 和指标观察热点与失败模式

### Q61. 为什么重试有时候很危险？

#### 回答要点

- 非幂等操作会被重复执行
- 下游已经高负载时，盲目重试会雪上加霜
- 如果没有退避和预算控制，会放大故障

### Q62. 幂等如何设计？

#### 回答要点

- 使用唯一请求 ID 或业务去重键
- 对写操作保存状态和执行结果
- 重试时能识别“已执行过”
- 明确哪些操作天然不幂等，需要额外保护

### Q63. 你怎么理解最终一致性？

#### 回答要点

- 在分布式系统中，短时间内不同副本或不同链路可能不一致，但在没有新写入后最终会收敛。
- 工程重点是定义可接受的不一致窗口和补偿机制。

### Q64. 限流、降级、熔断分别解决什么问题？

#### 回答要点

- 限流控制流量上限，保护系统
- 降级在资源不足时保核心路径
- 熔断在下游异常时快速失败，防止级联故障

## 12. Go / C++ / 语言基础

### Q65. goroutine 和线程是什么关系？

#### 回答要点

- goroutine 是用户态的轻量级协程，由 Go runtime 调度到线程上执行。
- 它不是操作系统线程，但最终仍依赖线程运行。
- 优势是创建和切换成本通常更低，适合高并发。

### Q66. `context.Context` 在 Go 中有什么用？

#### 回答要点

- 传递取消信号、超时和请求级元数据。
- 它是并发任务协作的重要基础，不是用来存任意业务对象的“大包袱”。

### Q67. 什么是 Go 中的逃逸分析？

#### 回答要点

- 编译器判断对象是在栈上分配还是逃逸到堆上。
- 逃逸可能增加 GC 压力，但不是绝对坏事，关键看是否合理。

### Q68. 虚函数调用在什么时候决定？

#### 回答要点

- 动态绑定在运行时决定。
- 编译期会生成通过虚表访问的机制，但最终调用哪个实现要看对象的实际动态类型。

### Q69. `vector` 扩容为什么会导致迭代器失效？

#### 回答要点

- 因为 `vector` 底层是连续内存，扩容通常需要重新分配更大的内存并搬移元素。
- 原先指向旧内存的迭代器、指针和引用就会失效。

### Q70. `map` 和 `unordered_map` 有什么差别？

#### 回答要点

- `map` 基于有序树结构，键有序，常见复杂度是 `O(log n)`。
- `unordered_map` 基于哈希，平均复杂度接近 `O(1)`，但无序且受哈希质量影响。

### Q73. 共享内存的原理是什么？

#### 回答要点

- 多个进程把同一段物理内存映射到各自虚拟地址空间。
- 它效率高，因为避免了大量数据复制。
- 问题是同步复杂，容易出现竞态和一致性问题。

### Q76. 为什么会跨域？哪些场景需要跨域处理？

#### 回答要点

### Q77. CSS 优先级怎么比较？

### Q78. 浏览器为什么有跨域限制，但后端服务之间没有这个问题？

### Q81. batch、吞吐、延迟三者之间是什么关系？

- 在线服务更关心尾延迟，离线批处理更关心整体吞吐。
- Agent 产品通常既要响应速度，也要成本可控，因此要平衡。

## 18. Transformer 架构深度

### Q93. MHA、MQA、GQA、MLA 的演进逻辑是什么？

#### 回答要点

- MHA：每个 head 独立 Q/K/V，KV Cache 最大。
- MQA：所有 head 共享一组 K/V，KV Cache 最小但质量下降。
- GQA：G 个 Q head 共享一组 K/V，平衡质量和效率（Llama 2 70B 使用）。
- MLA：低秩压缩 K/V 到 latent space，KV Cache 压缩约 93%。
- 演进动力始终是 KV Cache 的显存和带宽压力。

### Q94. RoPE 旋转位置编码的原理是什么？

#### 回答要点

- 对 Q 和 K 的相邻维度对施加旋转矩阵，旋转角度与位置成正比。
- 核心优势：相对位置编码——两个 token 的 attention score 只依赖相对距离。
- 外推性好：训练时见过的长度之外的序列也能处理（虽然效果会衰减）。
- 大多数现代 LLM 使用 RoPE（Llama、Qwen 等）。

### Q95. MoE（Mixture of Experts）在 Agent 场景中有什么特殊考虑？

#### 回答要点

- Agent 通常需要大模型做复杂推理，MoE 可以用更少的激活参数达到更大模型的效果。
- MoE 的推理延迟主要受 all-to-all 通信和 expert 权重加载影响。
- Agent 场景中 prefill 很长（system prompt + 工具描述），MoE 的通信开销占比更高。
- 路由策略直接影响不同任务类型的质量一致性。

### Q96. FlashAttention 为什么对 Agent 场景特别重要？

#### 回答要点

- Agent 的 system prompt 通常很长（工具描述、few-shot examples），attention 计算量大。
- FlashAttention 减少 HBM 访问，在长序列下加速效果更显著。
- Agent 的 decode 阶段也很长（多步推理），attention kernel 的调用频率高。
- 是推理引擎中 attention 后端的标配。

### Q97. PagedAttention 的 Copy-on-Write 机制是什么？

#### 回答要点

- 当多个请求共享相同前缀时，不复制 KV Cache block，而是增加引用计数。
- 只在某个请求要修改 block 时才真正复制（类似操作系统的 COW）。
- 在 Agent 场景中 system prompt 固定且长，COW 可以显著减少显存占用。
- 配合 Prefix Caching 使用效果更好。

### Q98. 位置编码外推（YaRN、PI）是什么？为什么 Agent 场景需要？

#### 回答要点

- 模型训练时的上下文长度有限，但 Agent 可能需要处理更长的对话历史。
- YaRN：对 RoPE 的频率做插值，让模型能处理更长序列。
- PI（Position Interpolation）：把位置索引压缩到训练范围内。
- Agent 场景中多轮对话累积的上下文可能超出训练长度，需要外推支持。

### Q99. Attention 的 O(N²) 复杂度对 Agent 有什么影响？

#### 回答要点

- Agent 的 system prompt 通常是几千到几万 token，加上多轮对话，N 很大。
- O(N²) 意味着 prefill 延迟随上下文长度平方增长，TTFT 可能达到数秒。
- 解决方向：FlashAttention（IO 优化）、滑动窗口、稀疏注意力、线性注意力。
- 在 Agent Infra 设计中，上下文管理直接影响用户体验。

### Q100. LayerNorm 和 RMSNorm 的区别？为什么 Agent Runtime 需要关注？

#### 回答要点

- LayerNorm：均值中心化 + 缩放。
- RMSNorm：只缩放（去掉均值中心化），计算更快。
- 大多数现代 LLM 使用 RMSNorm（Llama、Qwen 等）。
- Agent Runtime 做推理时需要确保 Norm 实现和训练一致，否则精度受损。

### Q101. 残差连接在深层 Transformer 中的作用？

#### 回答要点

- 缓解梯度消失，让信息能跳过中间层直接传递。
- Pre-Norm vs Post-Norm：Pre-Norm 训练更稳定，现代 LLM 普遍使用。
- 深层模型（100+ 层）可能需要额外的归一化策略。
- Agent 使用的大模型通常有 32-128 层，残差连接是训练稳定性的基础。

### Q102. SwiGLU 和 GELU 的区别？为什么现代 LLM 选择 SwiGLU？

#### 回答要点

- GELU：标准激活函数，近似 ReLU 的平滑版本。
- SwiGLU：引入门控机制（gate × value），表达力更强。
- SwiGLU 的 FFN 有 3 个 Linear 层（gate、up、down），参数量多 50%。
- 效果更好，现代 LLM（Llama、Qwen 等）普遍使用。

### Q105. Agent 评估框架和传统 LLM 评估有什么区别？

#### 回答要点

- Agent 评估框架：多轮交互 + 环境状态 → 动作序列 → 环境反馈/任务完成度。
- 需要沙箱环境（Docker/浏览器/API mock）保证可复现。
- 评估维度更广：规划、工具使用、反思纠错、成本效率。

### Q106. LLM-as-Judge 有什么缺点？怎么减少偏见？

- 减少偏见：交换位置多次投票、多 judge 模型投票、结构化评分标准（rubric）。
- 用人类标注校准 judge。
- 让 judge 先评价再比较，而不是直接比较。

### Q107. 如何保证 Agent 评估的可复现性？

#### 回答要点

- 固定随机种子、锁定模型版本。
- 容器化沙箱（Docker）固定 OS、依赖、工具版本。
- 固定数据集和配置版本。
- 记录完整 trajectory 和环境状态。
- 使用相同的 judge prompt。

## 20. Agentic 推理优化

### Q108. Agent 场景的 Prefill 和 Decode 有什么特殊性？

#### 回答要点

- Prefill 很长：system prompt（工具描述、few-shot）通常数千到数万 token。
- Decode 也很长：Agent 需要多步推理，每步可能生成几百 token。
- 前缀复用率高：同一 Agent 会话中 system prompt 不变。
- 对 TTFT 和 TBT 都敏感：用户等待时间直接影响体验。

### Q109. Prefix Caching 在 Agent 场景中怎么复用 system prompt？

#### 回答要点

- Agent 的 system prompt 通常固定且长（工具描述、规则、few-shot）。
- Prefix Caching 缓存 system prompt 的 KV Cache，后续请求直接复用。
- vLLM 的 APC 是 block-level，SGLang 的 RadixAttention 是 token-level（更精细）。
- 在前缀复用率高的场景下，TTFT 可以从 2.5s 降到 1.2s。

### Q110. DualPath 在 Agentic 推理中解决什么问题？

#### 回答要点

- PD 分离架构下，KV Cache 需要从 Prefill 节点传输到 Decode 节点。
- Agent 的 KV Cache 很大（长 system prompt + 多轮对话），传输带宽是瓶颈。
- DualPath 用两条路径并行加载 KV Cache，聚合带宽。
- 关键优化：分块传输、预取、与计算重叠。
