# 面试助手

一个零依赖的 Python 命令行脚本，用两份 Markdown 文件驱动面试准备：

- `plan.md`
- `question.md`

脚本文件为 `interview_assistant.py`，支持：

- 查看执行计划概览和某一天详情
- 按主题搜索题目
- 按天推荐练习题
- 按主题抽问
- 随机模拟面试
- 记录进度、自评分、卡点和复盘
- 自动把练习结果追加到当天 Markdown 日志

## 快速开始

在当前目录执行：

```bash
python3 interview_assistant.py
```

不带参数时进入交互式菜单。

也可以直接使用命令行参数：

```bash
python3 interview_assistant.py --help
python3 interview_assistant.py --overview
python3 interview_assistant.py --day 7
python3 interview_assistant.py --search Memory
python3 interview_assistant.py --practice-day 5 --count 3
python3 interview_assistant.py --topic MCP --count 5
python3 interview_assistant.py --mock --count 8
python3 interview_assistant.py --plan ./my_plan.md --question ./my_qa.md
```

## 工作方式

脚本启动后默认读取同目录下的两个文件：

- `plan.md`（可通过 `--plan` 指定其他路径）
- `question.md`（可通过 `--question` 指定其他路径）

并在当前目录下读写两个辅助文件：

- `.interview_assistant_state.json`
  - 持久化完成进度、复盘和练习历史
- `交互记录_YYYYMMDD.md`
  - 自动追加当天练习、自评分、卡点和复盘内容

## 脚本依赖的 Markdown 结构

这份 README 最重要的一点是：脚本并不是读取“自由格式”的 Markdown，而是依赖固定的标题层级和命名模式。

如果文档结构不符合约定，脚本可能：

- 找不到某一天
- 找不到题目
- 无法识别分类
- 搜索结果不完整
- 按天推荐的题目明显不准

下面分别说明两份文件的结构要求。

## 执行清单文件结构

对应文件：`plan.md`

### 解析规则

脚本会把每一个“天”的节点解析成一个 `DayPlan`。识别规则如下：

- 每一天必须使用二级标题 `##`
- 标题必须匹配下面这种模式

```md
## 3. Day 1：统一岗位叙事
## 4. Day 2: 项目深讲一，tagent
```

可以看到这里有几个关键约束：

- 必须以 `##` 开头
- 必须先有一个序号，例如 `3.`
- 必须包含字面量 `Day`
- 必须包含天数，例如 `Day 1`
- `Day N` 后面要接中文冒号 `：` 或英文冒号 `:`
- 冒号后面的内容会被当作这一天的标题

脚本当前使用的正则近似为：

```text
^##\s+\d+\.\s+Day\s+(\d+)[：:]\s*(.+)$
```

### Day 内部结构

在识别出一天以后，脚本会继续读取该天下面的三级标题 `###` 作为 section 名称，例如：

```md
## 3. Day 1：统一岗位叙事

### 今日目标

- 把"为什么是我、为什么是这个团队、为什么是这个岗位"讲顺
- 形成 90 秒自我介绍

### 今日任务

1. 写一版 90 秒自我介绍
2. 写一版 3 分钟岗位匹配表达

### 今日输出

- `90 秒自我介绍稿`
- `3 个项目的讲稿初版`
```

对脚本来说：

- `### 今日目标`、`### 今日任务`、`### 今日输出` 这些三级标题会被识别为 section
- section 下的所有普通文本、列表、空行，都会原样归到这个 section
- 四级及以下标题不会被单独建模，而是会被当作普通文本内容

### 推荐写法

虽然脚本对 section 名称本身没有强约束，但为了更好的可读性和复用性，建议每个 Day 都尽量保持类似结构：

- `### 今日目标`
- `### 今日任务`
- `### 今日输出`
- `### 验收标准`
- `### 晚间复盘` 或 `### 加练`

这样做的好处是：

- 用户查看某一天详情时结构稳定
- 不同主题迁移时复用成本低
- 后续如果要扩展自动分析，也更容易基于固定 section 名称做增强

### 最小可用模板

```md
# 某岗位 14 天执行清单

## 1. Day 1：主题名称

### 今日目标

- 目标 1
- 目标 2

### 今日任务

1. 任务 1
2. 任务 2

### 今日输出

- 输出物 1

### 验收标准

- 验收项 1
```

## 题库文件结构

对应文件：`question.md`

### 解析规则

脚本会把题库解析成一组 `Question`，并识别“分类 -> 题目 -> 题目内容”的层级结构。

#### 1. 分类层

分类必须使用二级标题 `##`，格式类似：

```md
## 2. 岗位匹配与动机
## 4. Agent 核心
## 5. MCP / Tool 平台
```

脚本当前使用的正则近似为：

```text
^##\s+\d+\.\s+(.+)$
```

也就是说：

- 必须是 `##`
- 必须以编号开头，例如 `2.`
- 编号后面的整段文字会被当作分类名

#### 2. 题目层

题目必须使用三级标题 `###`，并且符合 `Q数字.` 的格式，例如：

```md
### Q1. 你为什么适合这个团队？
### Q24. MCP 和普通函数调用相比，工程价值是什么？
```

脚本当前使用的正则近似为：

```text
^###\s+Q(\d+)\.\s+(.+)$
```

关键约束如下：

- 必须是 `###`
- 必须以 `Q1.`、`Q2.` 这种形式开头
- `Q` 后面必须是数字
- `.` 后面必须接一个空格，再写题目标题

#### 3. 题目 section 层

题目下面的四级标题 `####` 会被识别为这个题目的 section，例如：

```md
### Q1. 你为什么适合这个团队？

#### 回答要点

- 要点 1
- 要点 2

#### 一句话版

一句话答案。
```

对脚本来说：

- `#### 回答要点`、`#### 一句话版` 会被识别为 section
- section 下的文本和列表都会作为题目内容参与展示和搜索
- 没有放在 `####` section 下的内容，不会被结构化收集

### 推荐写法

为了兼顾练习体验、搜索效果和后续扩展，建议每道题尽量保持统一结构：

- `#### 回答要点`
- `#### 一句话版`
- `#### 追问点`
- `#### 常见误区`

其中当前脚本实际依赖最强的是：

- `分类名`
- `题目标题`
- `####` 下的内容

因为这些内容都会进入关键词搜索和按主题匹配。

### 最小可用模板

```md
# 某岗位高频问答题库

## 1. 分类名称

### Q1. 这里写题目

#### 回答要点

- 要点 1
- 要点 2

#### 一句话版

一句话答案。
```

## 为什么说“定义得好”会让脚本更泛用

这个脚本的泛用性，核心不在于内容属于哪个特定岗位，而在于"内容是否能被稳定结构化"。

换句话说，只要你把新的准备材料整理成同样的结构，这个脚本就可以迁移到：

- 其他 AI Agent 岗位
- 后端/基础架构岗位
- 研究工程师岗位
- 某个具体公司或部门的定制题库
- 证书考试、课程训练营、知识陪练场景

要提高泛用性，建议从下面几件事入手。

## 泛用化定义建议

### 1. 标题语义要稳定，不要频繁换格式

这是最重要的一条。

脚本依赖标题层级和命名模式做解析，所以同一类文档必须保持稳定格式。

建议：

- Day 标题始终写成 `## N. Day X：标题`
- 分类始终写成 `## N. 分类名`
- 题目始终写成 `### QN. 题目`
- 题目内容始终放到 `####` section 下

不建议：

- 有时写 `Day1`，有时写 `DAY 1`
- 有时写 `Q1:`，有时写 `问题 1`
- 把正文直接写在题目下而不加 `####`

### 2. 分类名和关键词要贴近检索词

脚本的搜索和按主题匹配，本质上是字符串包含匹配。

因此题库在定义时，最好让这些内容显式出现：

- 分类名
- 题目标题
- 回答要点中的关键术语

例如如果你希望通过 `--topic MCP` 找到一批题目，就应该让 `MCP` 明确出现在：

- 分类标题中
- 题目标题中
- 回答要点中

而不是只写一些模糊代称。

### 3. 每个 Day 的主题尽量能映射到一组题库关键词

脚本里的“按天推荐题目”依赖一个人工定义的 `DAY_TOPIC_HINTS` 映射，它会把某一天关联到一组关键词，再去题库里做匹配。

这意味着如果你想让“Day 7 的推荐题”更准确，需要满足两个条件：

- 执行清单的 Day 主题足够明确
- 题库中确实有对应关键词的题目

推荐做法：

- Day 标题写清楚主题边界
- 题库分类名与 Day 主题尽量用同一套术语
- 回答要点中显式出现这些主题词

例如：

- Day 写 `Memory / Context Engineering`
- 题库分类也使用 `Memory / Context Engineering`
- 题目要点里多次出现 `Memory`、`上下文注入`、`压缩`、`budget` 之类词

### 4. 一题尽量只表达一个核心问题

如果一个题目标题里混了多个问题，练习体验会变差，搜索也会更模糊。

推荐：

- 一题一个核心问题
- 追问单独放到 `#### 追问点`
- 不要把多个不相关问题塞进同一个 `Q`

这样对下面这些功能都有帮助：

- 随机模拟面试
- 主题抽问
- 自评分回顾
- 后续自动分析弱项

### 5. 内容要“可口述”，不只是“可阅读”

这个脚本不是文档浏览器，而是练习助手。

所以题库内容建议按“可回答”的方式组织，而不是写成长篇资料汇编。

更适合的写法：

- 先给定义
- 再给重要性
- 再给工程做法
- 最后给权衡和失败模式

不太适合的写法：

- 大段复制博客
- 只有概念，没有场景
- 只有结论，没有工程取舍

### 6. 统一 section 名称，为后续自动化预留空间

当前脚本虽然不会强依赖 section 名称做复杂逻辑，但统一命名可以显著提升后续可扩展性。

执行清单建议统一使用：

- `今日目标`
- `今日任务`
- `今日输出`
- `验收标准`
- `晚间复盘`

题库建议统一使用：

- `回答要点`
- `一句话版`
- `追问点`
- `常见误区`

这样后面如果要增加这些能力会更容易：

- 自动提取每一天的输出物
- 自动统计哪些题缺少“一句话版”
- 自动检查题库覆盖度
- 自动生成模拟面试脚本

## 迁移到其他主题时怎么定义文档

如果你要把这个脚本迁移到别的面试方向，不需要先改代码，先改内容结构就行。

例如你要做“后端架构师面试助手”，可以这样组织：

执行清单：

```md
## 1. Day 1：项目叙事与岗位匹配
## 2. Day 2：高并发与缓存
## 3. Day 3：数据库与事务
## 4. Day 4：微服务与治理
```

题库：

```md
## 1. 项目与经历
## 2. 高并发与缓存
## 3. 数据库与事务
## 4. 微服务与治理
## 5. 操作系统与网络
```

关键点不在于名字是什么，而在于：

- Day 主题能映射到题库分类和关键词
- 每个分类下的题目编号格式稳定
- 每个题目都有结构化 section

## 当前实现的边界

为了正确使用这个脚本，最好也了解它当前的边界。

### 1. 默认文件名可自定义

脚本默认读取：

- `plan.md`
- `question.md`

可通过 `--plan` 和 `--question` 参数指定任意路径的 Markdown 文件。

### 2. 解析是基于正则，不是完整 Markdown AST

这意味着它识别的是“标题行格式”，而不是通用 Markdown 语义。

所以：

- 标题格式要尽量严格
- 不要随意改变 `## / ### / ####` 层级
- 不要依赖花哨的排版技巧

### 3. 按天推荐是关键词匹配，不是语义检索

脚本现在不是 embedding 检索，而是简单关键词包含匹配。

因此题目是否会被推荐，很大程度取决于：

- 关键词是否真的写进题库
- 题库术语和 Day 主题是否一致

### 4. 题目正文必须放在 `####` section 下

如果你把内容直接写在 `### Q1. ...` 下面，而没有四级标题，当前实现不会把这些内容收集到题目结构里。

## 如果以后要进一步增强泛用性

这部分不是当前脚本必须做的，但如果你准备长期复用，建议后续考虑这些方向：

- 支持从配置文件读取 `DAY_TOPIC_HINTS`
- 支持更宽松的标题格式，例如 `Day1`、`D1`、`问题 1`
- 支持 Markdown AST 解析而不是正则匹配
- 支持标签字段，例如在题目里显式定义 `标签：MCP, Memory, Tool`
- 支持语义检索，替代纯关键词匹配
- 支持自动校验文档格式并输出缺失项报告

## 建议的编写检查清单

每次新建或修改计划/题库时，可以快速检查下面几项：

- 执行清单中的每个 Day 是否都是 `## N. Day X：标题`
- 每个 Day 下是否至少有一个 `###` section
- 每个分类是否都是 `## N. 分类名`
- 每个题目是否都是 `### QN. 标题`
- 每个题目下是否至少有一个 `####` section
- 重要关键词是否出现在分类名、题目标题或回答要点中
- Day 主题和题库术语是否一致

如果这些都满足，这个脚本通常就能稳定工作，而且迁移到其他准备主题时也会更顺手。
